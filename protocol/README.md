# Protocol

Shared compatibility manifests and cross-language fixtures will live here. The package remains pre-release and does not yet claim complete Turbo compatibility.

## State references

Component-action parameters resolve state references against the action's selected document, Frame, or form scope before Zod admission:

- `{"$state":"key"}` is an exact-value reference. It must contain only the `$state` key and may resolve to any value admitted by the action schema.
- `{{state:key}}` interpolates a scalar state value into a string. Structured values are rejected instead of being coerced to `[object Object]`.
- Arrays and plain objects resolve recursively. Object keys are never interpolated.

Missing/blank keys, malformed interpolation, ambiguous `$state` objects, cycles, unsupported object prototypes, and excessive depth are typed protocol failures. Resolution never falls through from a Frame/form scope to document or app-global state. Structured exact references must come from an explicit host codec; the package does not opportunistically `JSON.parse` arbitrary XML attributes.

## Native successful form controls

`FormControlRegistry` is the host-neutral bridge between app-installed native widgets and a logical form element. A registration is bound to the exact active control node and exact active form node, not only to a reusable key or XML `id`. Without a `form` attribute, the nearest ancestor form owns the control. A present nonempty `form="id"` overrides ancestry and must resolve case-sensitively to that exact unique ID in the active tree; blank, missing, or foreign owners fail closed. The host supplies current string values and updates them as native state changes. Headless callers must unregister and re-register after changing explicit ownership. Control removal/replacement unregisters that exact control; replacing/removing the form disposes the registry. A Turbo Stream child update preserves the form registry while disposing only removed controls.

The supported control descriptors deliberately cover only caller-admitted string-valued entries: one value, a hidden value, a checked/unchecked checkable, already-selected and enabled ordered multiple values, an explicit select snapshot, or a submitter. Collection omits disabled, unnamed, exact-empty-name, unchecked, and non-selected submitter controls. Owned entries follow whole-document XML order regardless of registration order or whether they appear before, inside, or after the form; duplicate names, duplicate values, and empty values are preserved. A hidden control with no supplied value and a checkable with no explicit value emit `""` and `on`, respectively. The caller activates one submitter with the selection token issued by its exact live registration; fabricated, foreign-registry, stale, non-submitter, disabled, and reassigned selections fail closed. The selected submitter is appended last with an empty default value, matching Turbo 8.0.23's `FormData`-then-submitter ordering. Rejecting a disabled submitter is an intentional native safety policy because ordinary native activation cannot originate from it; this does not claim browser-programmatic `requestSubmit` parity.

A select snapshot supplies options in authored order with current selectedness, direct disabledness, and whether each option belongs to a disabled `optgroup`. Each option supplies either an explicit string value or a raw host text snapshot; an explicit value, including `""`, wins, while fallback text is stripped and collapsed using only HTML ASCII whitespace. A disabled or unnamed select emits nothing; otherwise each selected option that is neither directly disabled nor inside a disabled `optgroup` emits one entry in option order, preserving repeated and empty values. The host owns current selectedness, option order, group state, and the text snapshot; core does not discover `<select>`, `<option>`, or `<optgroup>` nodes or choose a default option.

An explicit hidden descriptor admits any name/value, defaults an absent value to `""`, and replaces only an ASCII-case-insensitive `_charset_` name with `UTF-8`, matching the browser hidden-input entry-list branch used by Turbo's `FormData(form)`. Authored name casing and duplicate sentinels are preserved in document order. Ordinary value descriptors named `_charset_` are not rewritten, and the manually appended submitter keeps its raw value. A value or hidden descriptor may carry one admitted directionality snapshot with a nonempty companion name and an exact host-resolved `ltr` or `rtl` value. That frozen companion follows its admitted primary value immediately, including an empty value or hidden `_charset_` replacement; disabling or omitting/emptying the primary control name suppresses both entries. The host owns eligible native-control selection and must update the current value, companion name, and resolved direction together. Core does not discover hidden inputs or read XML `dirname`/`dir`, text, locale, Unicode heuristics, or React Native layout direction to infer browser directionality.

Collection returns frozen name/value entries, not a complete HTML entry list, query string, `FormData`, or request body. Turbo 8.0.23 constructs `FormData(form)` without the activated submitter and then manually appends only its nonempty name plus raw value or `""`; this package therefore deliberately emits neither browser image-button `.x`/`.y` coordinates nor submitter `dirname` sidecars. App-installed registered components plus `useExpoTurboFormControl` are the native equivalent of form-associated components for the supported string descriptor families; browser custom elements and `ElementInternals` callbacks are N/A, while their arbitrary multi-name or File/Blob entry values remain unsupported with uploads. Automatic XML/DOM control discovery, browser select defaulting, native files, automatic or inherited `dirname` resolution, and complete eligible-control coverage remain pending. Constraint validation/focus, lifecycle events and request-mutation hooks, physical accessibility evidence, and morph-aware `422` field-state preservation are also separate unsupported contracts.

`DocumentFormControls` is the host-owned document-lifetime collection for these registries. Repeated lookup returns the same registry while the exact form node remains active, including a child-only Stream update. Form removal or replacement disposes that identity; a same-key replacement receives a fresh registry. Explicit collection disposal releases every owned registry. React never disposes this collection. Instead, every mounted `ExpoTurboFormScope` retains the session-shared exact-form activity and defers release by one microtask; development Strict Mode replay or another provider can reacquire before release, while the last real form-scope/provider unmount cancels the exact pending confirmation or active request. Headless callers retain explicit `cancelSubmission` ownership. The provider separately reference-counts its injected document state/scope owners across effect replay and disposes them after their last real unmount.

An app-installed form component declares `formOwner: true` in its registry definition and mounts `ExpoTurboFormScope`, which also supplies the existing form state scope when configured. Descendant native components call `useExpoTurboFormControl` with their current admitted descriptor. A registered component with `form="id"` may instead call `useExpoTurboFormControl`, `useExpoTurboForm`, or both outside that form's React subtree; the exact declared owner overrides the nearest `FormContext`. The renderer subscribes above the component error boundary to the owner key, so insertion, removal, and same-ID replacement retry the association and rebind surviving controls to the fresh exact registry. A target that is missing, blank, or not declared with `formOwner: true` fails closed. Association changes form collection/submission ownership only and never moves the component into the target form's React or state-scope ancestry.

Registration follows the exact logical control node, updates in a layout effect without registration churn, unregisters on React cleanup, and binds to the resolved exact form. `useExpoTurboForm` exposes the stable form identity, frozen successful-entry collection, exact active-form request planning, atomic destination-aware submission proposals, observable submission state/busy accessibility, frozen terminal state, and the canonical submit/cancel/dismiss/safe-retry paths. `useExpoTurboFormControl().selection()` returns the active registration-bound token passed to `successfulEntries`, `requestPlan`, or `submissionProposal` and fails after cleanup or owner rebinding; that control binding also exposes the activated submitter's pending/effective-disabled/submits-with presentation described below. Proposal re-admission rechecks that an exact external submitter still owns the captured form, so reassignment during async confirmation cannot fetch under the former owner. Rendering either hook without an active nearest or explicit declared owner fails through the registered component error boundary. Planning and proposal creation remain pure; the form binding delegates submission ownership to the canonical core controller. Validation remains a separate contract.

Component definitions may declare `formContainer: "fieldset"` or `formContainer: "legend"`; the role is capability-visible and aliases resolve to the canonical definition without decoding props. Passing that component registry through `DocumentFormControls` as `formSemantics` enables native fieldset disabledness for explicitly registered controls. Core reads raw `disabled` attribute presence from every current semantic-fieldset ancestor and omits the control unless it is below that fieldset's first direct semantic-legend child. Every ancestor is evaluated independently, text/comments/non-legend siblings do not change which legend is first, nested legends are not direct legends, and explicit `form` ownership never changes the control's physical ancestry. Direct disabledness still wins. Activated submitters inherit the same rule and are rejected before planning, while `useExpoTurboFormControl` subscribes to semantic-fieldset ancestors and projects direct, inherited, and pending disabledness through its native accessibility binding. The registry does not discover controls or infer browser `<fieldset>`/`<legend>` tags automatically.

Component definitions may also declare `formContainer: "datalist"`. Every explicitly registered control physically below a semantic datalist is omitted before ordinary descriptor-specific entry construction, including value, checkable, select, hidden, directionality, and externally form-owned controls. Turbo 8.0.23 constructs browser `FormData` without a submitter and then manually appends the selected submitter, so this package deliberately preserves that append-last submitter behavior even when the submitter is physically below a datalist; this differs from the current HTML optional-submitter entry-list algorithm. Datalist ancestry is evaluated independently from disabled fieldsets and does not make the control disabled, so `useExpoTurboFormControl` continues to expose only direct, fieldset-inherited, and pending disabledness. The registry does not discover or infer browser `<datalist>` tags, supply suggestions, or hide datalist children automatically; the host component owns those presentation choices.

`DocumentFormControls` accepts a Turbo-compatible `formMode` of `on` (the default), `off`, or `optin`. `FormControlRegistry.shouldInterceptSubmission` and its React binding are synchronous interaction-boundary queries: `off` never intercepts; `on` requires the form and activated submitter, when present, to each have no closest `data-turbo="false"`; `optin` requires the submitter to remain navigatable and the form or any form ancestor to carry exact `data-turbo="true"`. Internal and external submitters use their own ancestry independently from the associated form, values are case-sensitive, and attributes are read live on every call. The explicit `requestPlan`, `submissionProposal`, `submit`, and retry APIs remain host-owned bypasses, matching Turbo's separation between observer interception and imperative navigation.

## Native form request planning

`buildFormRequest` is a pure boundary between collected string entries and a future form-submission controller. It accepts the owning response's final document URL, raw form and activated-submitter attributes, already-collected entries, and protocol request metadata. A named submitter must match the final collected entry exactly, preserving the registry's Turbo-compatible append-last contract without introducing a second value source. The planner resolves an explicitly present submitter action before the form action, resolves relatives against the document URL, and admits only credential-free same-origin HTTP(S). Any explicit action fragment delimiter, including an empty `#` or `?#`, fails until native navigation owns anchor behavior; it is never silently dropped.

Source method and enctype use submitter-before-form precedence. Method matching is case-insensitive for GET, POST, PUT, PATCH, and DELETE; an unsupported source method follows Turbo 8.0.23 and becomes GET. Enctype matching admits URL-encoded, multipart, and `text/plain`; an unsupported value becomes URL-encoded. Empty submitter method/enctype values fall through to the form. Attribute presence on either owner opts a GET into Turbo Stream responses, while every unsafe source method requests Streams automatically.

GET replaces the action's complete existing query with the ordered entries, emits no body, ignores enctype, and does not interpret `_method`. Unsafe actions preserve their query and use Rails transport normalization: activated submitter `_method`, submitter method, first `_method` entry, then source method determine the effective method. Effective POST removes all `_method` entries; PUT/PATCH/DELETE collapse them to one lowercase override at the first occurrence or append one, and the wire method is POST. Blank, GET, and unsupported unsafe overrides fail closed.

Names and values normalize CR/LF to CRLF before query/body encoding. The planner treats already-materialized `_charset_` and directionality entries as ordinary ordered records and does not rediscover their semantics; the documented Rails normalization still applies when any entry uses the reserved `_method` name. URL-encoded bodies use ordered `URLSearchParams` serialization and `application/x-www-form-urlencoded;charset=UTF-8`. Unsafe `text/plain` fails until the gem supplies an explicit decoder because Rack does not expose those fields or `_method` as form parameters; claiming an effective Rails verb before that contract exists would be false. Unsafe multipart likewise fails until an injected native upload/boundary adapter exists. The returned source/effective method, encoding, entries, request, headers, and body records are cloned and frozen; protocol header metadata rejects control characters, and the planner owns no fetch, cancellation, pending UI, or response state.

`FormControlRegistry.requestPlan` composes that pure planner with the exact active form identity. It reads the form's current raw `action`, `method`, `enctype`, and `data-turbo-stream` presence; collects current registered values in XML order; and, when the caller supplies a registration-bound submitter selection, reads that node's current `formaction`, `formmethod`, `formenctype`, and Stream-marker presence while taking its name/value from the live registered descriptor. The caller owns the request ID, optional capability metadata, and abort signal; active-form callers cannot inject `frameId`. The same method is exposed by `useExpoTurboForm`. Composition is synchronous and immutable: it performs no fetch, form/Frame destination-target selection or dispatch, pending-state change, validation UI, or response handling, and a removed/replaced form or submitter fails through the existing exact-identity guards.

## Native form submission proposals

`FormControlRegistry.submissionProposal` atomically combines one exact live form/submitter snapshot with destination resolution and request planning. Logical Frame targeting uses `data-turbo-frame`; HTML `target` and submitter `formtarget` are browser browsing-context features and fail closed, including when present with a blank value. The activated submitter has precedence over the form. Inside a Frame, the first present submitter/form value then falls through to the enclosing Frame's `target` and current Frame according to Turbo 8.0.23: a blank submitter masks the form value, `_self` stays current, `_parent` selects the immediate enabled parent or document, `_top` selects the document, a missing named target stays current, and a disabled current/named/parent target selects the document. Outside a Frame, a blank submitter falls through to the form value; only an active enabled named Frame is captured, while absent, missing, disabled, `_top`, `_self`, and `_parent` targets remain document submissions.

The frozen proposal contains a public frozen destination plus the planner-issued request plan. Frame destinations derive the `Turbo-Frame` request header internally; document destinations omit it. A hidden admission record retains the issuing session, tree generation, exact form, activated submitter, exact destination Frame identities, and the raw confirmation message resolved at that same snapshot. Submitter `data-turbo-confirm` wins by attribute presence; an empty value masks the form's value, whitespace is preserved, and an absent submitter value falls back to the form. This hidden state lets the next ownership layer reject same-key replacements and even an old tree object restored after a full document replacement without putting confirmation metadata into the request. The same proposal API is exposed by `useExpoTurboForm`. Proposal creation performs no fetch, cancellation, epoch acquisition, request supersession, pending UI, or response application.

## Native form transport and response disposition

`FormRequestExecutor` remains the explicit transport-only API. It owns one caller-scoped lane and returns classified immutable candidates without resolving a destination or mutating the session. `execute` first creates an `AbortController` and passes its exact signal to a synchronous `FormRequestPlanFactory`. The returned plan must retain that signal and the immutable shape produced by the planner. Only after a new plan is admitted does it abort the previous execution in that executor, so an invalid plan or thrown planning error cannot cancel valid in-flight work. Separate executor instances remain independent.

`FormSubmissionController` is the canonical destination-aware transport-and-application entry point. `submit` creates the signal first, requires the synchronous factory to return the exact session-issued `FormSubmissionProposal`, and re-admits its planner-issued plan and signal. When the hidden proposal contains confirmation metadata, a configured `FormConfirmationAdapter` receives exactly that string plus the owning abort signal and must resolve to a literal boolean. This is an intentional fail-closed native adapter contract; unlike Turbo's browser callback, other truthy values are rejected. Missing configuration fails before displacing admitted work. The valid adapter runs under a hidden exact-form owner: the public activity snapshot remains idle, the submitter remains enabled, and no destination is claimed or fetched while it is pending. Denial returns the existing frozen canceled report. Acceptance rechecks the exact form, submitter, destination Frame, and tree-generation identities before destination ownership and fetch; cancellation settles even when an adapter ignores its signal, and a late answer or rejection is inert.

Document submissions share one session-scoped lane with every `DocumentRequestLoader`; Frame submissions share the lane indexed by Frame ID while retaining the exact Frame node with every `FrameRequestLoader`. Any admitted request to the same destination supersedes its predecessor across GET and form transports. For the same exact form, `duplicateBehavior` defaults to `prevent`, including while confirmation is pending, while an explicit `supersede` admits a replacement even when it changes destinations. A confirmation-bearing explicit supersede clears the displaced request's busy/submitter presentation before awaiting the replacement answer; denial does not restore the displaced request. Different forms in different Frames and the document/Frame lanes remain concurrent. A Frame named `document` is still distinct from the document lane.

Each exact form registry publishes a stable `FormSubmissionActivitySnapshot` through `submissionState` and `subscribeSubmission`; every controller for the same `DocumentSession` observes the same exact-form activity owner. Default duplicate prevention is atomic: once a submission becomes active, another rapid submission for that exact form is prevented before it can fetch or claim a destination. The explicit `supersede` policy installs the replacement activity before aborting the incumbent, and exact-owner settlement prevents a stale abort/finalizer from clearing the newer snapshot. `cancelSubmission` cancels only the exact active owner, and form removal/replacement cancels that owner during exact-node disposal.

The active snapshot exposes busy state, request ID, and exact activated-submitter key for the immutable planner-issued request used by transport. Only that submitter receives a `FormSubmitterActivitySnapshot`: `pending` is true and `submitsWith` exposes a nonempty raw `data-turbo-submits-with` value without mutating the XML node or request. React projects these values through the form binding's `state`/busy accessibility; the control binding exposes pending/submits-with and derives effective disabled/accessibility state from authored disabled or pending. Unrelated controls and forms remain unchanged.

Activity cleanup is deterministic and precedes document, Frame, or Stream response application. Confirmation denial/failure, success, classified HTTP error, transport or parsing failure, explicit abort/supersession, form or submitter removal, and final mounted React form-scope/provider unmount all release the exact owner once; a stale confirmation or completion cannot clear or revive a newer explicit supersede. Scope ownership is keyed by the session-shared exact form rather than a registry or collection, so Strict Mode replay and multiple providers cannot cancel an incumbent prematurely. The deferred React release creates an intentional one-microtask cancellation window; headless submissions require explicit cancellation.

The same exact-form owner publishes a separate stable `FormSubmissionTerminalSnapshot` only after request activity and activated-submitter presentation have cleared. It distinguishes `applied`, `empty`, `canceled`, uncommitted `failed`, and `committed-error` outcomes without retaining request/response bodies, field values, confirmation text, URLs, raw host causes, or server-authored identifiers from parser/action messages. Applied snapshots retain only the method, request ID, response classification/status, application kind, and optional submitter node key. Failure snapshots use a static package-owned error name/message, allowlist only a valid numeric response status and parser location, and carry one of `safe`, `unsafe`, or `committed` retry dispositions. A valid applied `422`, `4xx`, or `5xx` XML response is therefore an applied server-owned result, never a generic client failure surface. A mutation followed by finalization failure remains `committed-error` so a host cannot accidentally present it as safely replayable.

Terminal settlement is gated by the newest admitted exact attempt. Duplicate attempts that never acquire form activity, stale superseded completions, removed/replaced forms or submitters, and late confirmation answers cannot overwrite the current snapshot or leak into a same-key replacement. Starting accepted work clears an older terminal result; a denied or otherwise canceled pre-start confirmation leaves the prior useful result visible. Exact-node disposal clears the result and invalidates recovery. React hosts may inject `formComponent` into `ExpoTurboProvider`; `ExpoTurboFormScope` renders that stable boundary around the still-mounted native fields and supplies the same binding as `useExpoTurboForm`, so result UI does not require replacing the form subtree.

`retryFailure` is explicit recovery, not automatic request replay. It is available only for an uncommitted failure marked `safe`: failure before fetch invocation or any uncommitted GET failure. A host must provide a request ID different from the failed attempt. The registry reuses the exact still-live submitter registration, recollects current control values and current form/submitter attributes, reruns confirmation and normal admission, and sends at most one newly planned request. Once a non-GET fetch was invoked the failure is `unsafe`, because the server may already have processed the mutation; a committed finalization error is `committed`. Both reject retry before fetch and require product-specific reconciliation.

An optional host `formAnnouncements` adapter consumes only the redacted non-`none` terminal snapshot plus its exact form key. React baselines the current revision on mount and claims each later revision once against the exact logical form, only after the active form has returned to idle. Initial or cleared states, pre-start denial, stale or superseded work, exact-node disposal, same-key replacement, and final-scope unmount do not announce; Strict Mode replay and another provider cannot claim the same revision twice. The package supplies no user-facing copy and imports no platform accessibility API: the host maps the structured snapshot to localized text and platform delivery. Delivery failure reports through the provider error callback without changing the settled submission outcome or blocking later revisions. Constraint validation/focus, a form lifecycle event family, mutable before-request hooks, physical accessibility evidence, form-driven snapshot invalidation, and morph-aware validation preservation remain unsupported.

Non-canceled response reports expose `responseUrl` as the validated final transport URL. It is evidence about the response, not an assertion that the active document URL or a Frame `src` adopted it. Applied document and Frame reports identify the actual `applicationDestination`; cross-target Frame errors can therefore report requested destination B and application destination A without ambiguity. Applied document and direct Stream reports expose `streams.interrupted`, while applied Frame reports expose `frame.streams.interrupted`; empty and canceled reports expose neither.

URL, first request-ID/header, exact Frame, proposal, plan, and signal admission all happen before a claim displaces its incumbent. Claims install the new lease before aborting old work, then recheck ownership before fetch, so reentrant abort listeners may start still-newer work without allowing the stale outer claimant to fetch. Owner-aware stale cancellation and settlement release only their exact lease and cannot clear or abort a newer owner. Ownership is scoped to the exact `DocumentSession`; Frame recurse requests retain the original target Frame lane even while an intermediary Frame ID is sent in a later request header.

The fetch adapter receives the plan's exact request object once, including its wire method, headers, body bytes/string, body content type, request ID, and abort signal. The executor retains ownership through both `fetch` and the single response-body read. Explicit cancellation or a newer admitted execution suppresses a late fetch/body resolution or rejection and returns a frozen `canceled` report instead of a network error. A current adapter or body-read failure becomes a redacted `RequestError`; canceling transport cannot prove that an unsafe request was not already processed by the server.

After the adapter follows redirects, the executor requires a nonblank credential-free same-origin HTTP(S) final URL and derives redirect metadata from either the adapter flag or a changed final URL. Raw `3xx` responses reject. A `204` is an `empty` candidate without reading a body; a blank `201` is also `empty`. Every other candidate requires either `application/vnd.expo-turbo+xml` or `text/vnd.turbo-stream.html`, buffers the body exactly once, and reports `xml` or `stream` plus `success` (`2xx`), `client-error` (`4xx`), or `server-error` (`5xx`). A Stream response is admitted only when the immutable plan negotiated it. Missing, unsolicited Stream, JSON, and HTML MIME types fail closed. `xml` deliberately does not claim whether the later dispatcher will treat the body as a document, matching Frame, or validation fragment.

The controller retains the exact destination lease through response parsing and rechecks both that lease and the hidden proposal identity immediately before the first mutation. Staged Stream dispatch also checks that the lease is still retained before each sibling action and built-in target. If synchronous notification or host action starts newer work, already committed mutations remain visible, undispatched work stops, `streams.interrupted` is true, and the newer owner remains authoritative. A `canceled` result and an ordinary `204` or blank `201` are no-ops. A negotiated Stream response dispatches once through the shared Stream dispatcher for `2xx`, `4xx`, or `5xx` without document or Frame extraction. Fatal Stream parse failures mutate nothing; per-action protocol failures retain the dispatcher's existing isolated, continue-siblings behavior, while post-commit session-finalization failures remain observable as committed outcomes.

Expo XML success applies to the frozen destination. A failed submission from one Frame to another instead transfers ownership to, and extracts, the exact origin Frame, matching Turbo 8.0.23; a top-level form targeting a Frame still uses that target because it has no enclosing origin Frame. Frame application preserves the mounted wrapper and consumes embedded Streams. Its canonical `src` changes for a successful nonempty Expo XML response, an adapter-followed Expo XML response, or a redirected `204`/blank-`201`; Stream responses and unredirected `4xx`/`5xx` or empty responses leave the existing `src` unchanged. A document response parses before replacement. Safe GETs, nonempty `201`, adapter-followed final bodies, and well-formed `4xx`/`5xx` documents are authoritative; only an unsafe nonredirect document `200` fails the Turbo redirect-safety rule. Successful document XML owns the response's validated final URL, including after an adapter-followed redirect, while every top-level `4xx`/`5xx` replacement preserves the active document URL. The adapter has already followed any redirect and supplied the final buffered body, so disposition never issues a second GET.

Every owned XML error document is applied with its `client-error` or `server-error` classification. A `422` therefore renders authoritative server values by replacement, but this does not claim morphing or preservation of local native input state. Malformed XML, a missing matching Frame, wrong MIME, transport failure, or any stale form/submitter/destination/tree-generation identity fails or cancels without mutation. The activity contract above clears pending state before any response application and the separate terminal contract records only the newest surviving exact-form outcome. Form-driven snapshot invalidation and morph-aware validation preservation remain unsupported.

## Semantic styles

Styling is component-schema-owned. A component may bind an explicit whitespace-separated `style-tokens` attribute with `tokenListCodec`; XML `class` remains structural selector metadata and is never forwarded as native `className`.

A host `StyleAdapter` exposes a frozen deterministic token manifest and resolves only app-compiled semantic definitions. Components create a style hook from that exact adapter with `createComponentStyleHook`, which binds the native style type and verifies the provider uses the same adapter. Resolution uses the component's canonical registered tag, including when XML uses an alias, and rejects unknown, repeated, excessive, same-group conflicting, and component-inapplicable tokens. The standalone Expo corpus demonstrates tone, spacing, layout, safe-area, and host-platform variants without accepting arbitrary React Native style objects or dynamic Tailwind/NativeWind values.

Style composition is deterministic: component defaults are lowest priority, admitted independent token groups compose in authored order, explicit schema-owned semantic props override tokens, and local transient native state such as pressed/disabled remains the component's highest-priority layer. Unsupported input raises a typed props failure inside the registered component error boundary; it is never ignored or normalized into a fallback class.

## Mutation identity and cleanup

Stable XML IDs provide addressability; they do not make two different tree nodes or document sessions the same runtime instance. A Turbo Stream `update` preserves its target node, component state, Frame controller, and target-owned scope while replacing children. `replace` and append/prepend/before/after ID-collision replacement create a new node identity even when the authored ID and tag are reused. React remounts that instance, old disposal hooks run exactly once, and a same-ID Frame wrapper cancels its old request/controller before the replacement can connect. Frame requests also carry exact node and controller ownership, so a retained stale controller can neither cancel a replacement request nor commit a late response into the replacement wrapper.

Turbo 8.0.23 selects structural morphing only when a `replace` or `update` Stream carries the exact lowercase `method="morph"`. The native dispatcher isolates that action as an error before target resolution or mutation until a native reconciler exists. Missing, blank, differently cased, and arbitrary method values retain ordinary plain `replace`/`update` behavior; they are not rejected or normalized into morphing.

Built-in compound Stream actions validate every payload ID, including repeated IDs within one template, before removing collision nodes. A rejected action therefore leaves the active tree, revision, snapshots, scopes, and disposal registrations unchanged while later sibling Stream actions may continue. This preflight guarantee applies to the built-in dispatcher; arbitrary host mutations and custom Stream handlers must not mutate and then throw as though they were transactional.

Registered component cleanup runs on logical subtree removal/replacement and on ordinary React/provider unmount. Cleanup is identity-bound and once-only, so a logical mutation followed by React reconciliation cannot dispose the same resource twice.

## Document GET transport

`DocumentRequestLoader` is a host-neutral transport and tree-replacement primitive for a mounted `DocumentSession`; the observable lifecycle controller is layered on it below. It is not a navigation, history, or cache controller. It resolves relative sources against the active document URL and permits only credential-free, same-origin HTTP(S) request and final-response URLs. Document GETs advertise the Expo Turbo MIME type and send protocol, runtime, request-ID, and optional capability headers without a `Turbo-Frame` header.

After the fetch adapter follows redirects, the response's final URL becomes the parsed document URL. Valid XML responses are classified as `success` (`2xx`), `client-error` (`4xx`), or `server-error` (`5xx`) and replace the active tree only after MIME validation and a complete parse. A valid `4xx` or `5xx` XML document still commits as authoritative server output while retaining its distinct classification for the visit controller. An optional synchronous `beforeCommit` admission callback receives only frozen response metadata, candidate status, and the candidate document's normalized root location; it can commit or discard the candidate but cannot access or mutate the private parsed tree. An empty candidate reports the active document's root.

A `204` response and a blank `201` response are deliberate native empty outcomes that preserve the current tree and URL; they are not a claim of exact browser Drive GET behavior. Raw `3xx` responses, missing or cross-origin final URLs, wrong MIME types, malformed XML, and transport failures reject with typed errors without replacing the current tree.

Explicit cancellation, supersession by a newer document GET or document-destination form request from any loader/controller instance, or any intervening session-tree replacement prevents the stale response from committing and produces a canceled outcome. Tree ownership uses a monotonic generation, so restoring an earlier tree object does not revive an earlier request. `cancel(owner)` affects only a request started with that exact owner; stale owner cancellation cannot cancel a form request that has since claimed the document lane. Disposal failures raised after tree replacement remain visible even though the new tree already owns the session.

The raw loader does not decide navigation actions, root visitability, lifecycle events, snapshot/history restoration, preload/prefetch, or form submission. The separate visit controller below owns the currently supported root policy plus loading and accessibility state.

## Document visit lifecycle and native accessibility

`DocumentVisitController` gives one mounted document session an identity-safe observable GET lifecycle. Its immutable snapshot moves from `initialized` to `started`, then to exactly one of `completed`, `failed`, or `canceled`. `busy` becomes true immediately at `started`. `progressVisible` becomes true only after the injected clock reaches `DOCUMENT_VISIT_PROGRESS_DELAY_MS` (500 ms) and is cleared on every terminal path.

A committed success document, `204`, or empty `201` completes the visit. A valid `4xx` or `5xx` XML error document commits first and then fails the visit. Network, MIME, parse, and pre-commit policy errors fail while preserving the prior tree. Explicit cancellation and supersession are `canceled`, never request errors. Epoch, request-owner, and session-generation checks prevent stale requests or cleared timers from publishing into a newer visit. A session finalization error after a successful commit remains observable while the lifecycle records `completed`; the equivalent error after an authoritative `4xx`/`5xx` commit records `failed`.

The sole XML root may declare `data-turbo-root`; absence means `/`, and relative values resolve against that document's URL. Top-level visitability follows Turbo 8.0.23's exact origin-plus-path prefix algorithm and fixed case-sensitive unvisitable-extension set. A cross-origin proposal delegates to `NavigationAdapter.openExternal`; a same-origin proposal outside the root or carrying an excluded extension delegates to `NavigationAdapter.visit` with `advance`. Explicit initial `replace` and `restore` visits reject before request ownership changes. Admission and host delegation happen before document request ownership changes, so an unvisitable proposal does not cancel an active visit.

After a successful same-origin redirect, the controller reclassifies the final URL against the parsed response document's root before tree replacement. An unvisitable candidate is discarded and the final URL is delegated with `replace`; a visitable candidate commits. A nonredirect response may redefine its root and commit normally, while redirected `4xx`/`5xx` XML remains authoritative error output and is not reproposed. This is a native-equivalent ordering: unlike browser Turbo, the discarded response is not rendered before host delegation, so adapter failure leaves the prior tree active. Cross-origin final redirects remain a typed transport rejection because the loader refuses to admit a final URL on another origin.

React hosts inject the controller through `documentController`, may supply a `documentComponent` boundary, and can read the nearest binding with `useExpoTurboDocument()`. The binding exposes the frozen controller snapshot plus a frozen `{ busy }` accessibility state for an actual native boundary. Controller ticks preserve the boundary and do not remount unchanged registered children. The controller remains host-owned across React unmount and Strict Mode effect replay; React subscribes and reports errors but never silently cancels it. State and error observers are snapshotted and isolated; hosts can provide `onObserverError` to surface observer failures without changing the visit's request result, and an unhandled observer failure is rethrown on a microtask.

The controller currently accepts only initial `advance` visits and uses `replace` solely when delegating an unvisitable successful redirect. It does not yet own router URL/history state or restoration identifiers. Cache previews/restoration, lifecycle-event parity, document-visit live announcements, browser progress-bar animation, and physical-device accessibility evidence remain unsupported.

## Document refresh Streams

The built-in `refresh` Stream action ignores `target`, `targets`, and template content. It captures the active document URL plus optional `method`, `request-id`, and `scroll` attributes, then delegates to the host-wired `DocumentRefreshController`; dispatch itself never performs a nested fetch while a Stream response still owns its destination lease. A missing controller or active document URL is an isolated action error and does not prevent later sibling Stream actions from running.

Turbo 8.0.23 selects refresh morphing only for exact lowercase `method="morph"`; an absent, blank, differently cased, or arbitrary method value uses ordinary replacement. The native controller therefore rejects only that exact morph request, plus every explicit scroll policy, until native morph and scroll contracts exist. Accepted requests use Turbo's 150 ms trailing debounce. At execution time, only the newest pending request may proceed, its captured URL must still exactly own the active document, and an already-started document visit is never interrupted. The refresh then reuses the ordinary document GET loader and observable visit lifecycle to replace the complete document from canonical truth.

Each document session retains the newest 20 request IDs in insertion order. Document GETs, Frame and recurse GETs, and canonical form submissions record their generated ID immediately before transport; a caller-scoped `FormRequestExecutor` can be given the same tracker explicitly. A refresh carrying one of those IDs is suppressed when the debounce fires, preventing the originating request from refreshing itself twice. This is loop suppression, not request replay, cache restoration, morph preservation, history synchronization, or missed-Cable-message recovery.

## Document-tree link activation

`useExpoTurboDocumentLink(href)` lets an app-owned registered component expose a plain native link without adding React Native or a host router to the package. The hook requires the host-injected `DocumentVisitController` and returns a stable activation callback without subscribing the component to visit-state ticks. An ordinary same-origin top-level activation delegates the unchanged `href` to that shared controller, so request headers, cancellation, response classification, and latest-visit ownership use the document GET contracts above. A link-only resolver separately admits normalized `mailto:` and `tel:` URLs; the HTTP(S)-only resolver used by document, Frame, and form fetch paths is unchanged.

Unless the closest `data-turbo` setting opts out, every HTTP(S) interactive link is classified by the active document root before Frame capture. Cross-origin, outside-root, and excluded-extension links delegate through the document visit path even when authored inside a Frame. A visitable activation inside a Frame then passes the unchanged `href`, nearest Frame ID, and optional `data-turbo-frame` value to the host-injected `FrameControllerCollection.visit` contract. That is the same registry-owned path as a programmatic Frame visit, so Frame-default/element target precedence, current/named Frames, `_self`, `_parent`, `_top`, URL admission, controller reuse, request cancellation, and response ownership cannot drift into a second React implementation. A missing named target inside a Frame falls back to the current Frame; a disabled current/named Frame or unavailable `_parent` promotes through the shared document coordinator. Outside a Frame, a non-`_top` target identifying an active enabled Frame enters the shared Frame visit path and applies its target rules. Top-level `_top`, missing, and disabled targets follow ordinary top-level activation. Frame and document controllers keep independent request ownership.

Hosts inject `navigation` into `ExpoTurboProvider`; ordinary root/cross-origin activation passes that adapter into the host-owned document controller, while explicit opt-out uses it directly. A credential-free cross-origin HTTP(S) URL awaits `openExternal` with the normalized absolute URL. Normalized `mailto:` and `tel:` URLs also await `openExternal`, return a frozen scheme-identifying result, and delegate before document/Frame target capture, so top-level, Frame-contained, named-Frame-targeted, and opted-out links share one external path. The closest `data-turbo` setting wins for HTTP(S): exact `false` delegates a same-origin URL through `visit(url, "advance")`, delegates a cross-origin URL through `openExternal`, and a nested non-`false` setting restores normal Frame or document capture inside an opted-out ancestor. The Frame registry receives the same `NavigationAdapter` for external HTTP(S) visits and the shared document controller for same-origin `_top` or promoted visits. Successful delegation returns a frozen `status: "delegated"` result identifying external, scheme, or owning-host navigation. A missing or failing adapter rejects the activation Promise and never starts, supersedes, or cancels a document request.

The activation callback is bound to the exact registered protocol node that created it. It rechecks node identity at press time and fails closed after replacement/removal even if a deterministic key is reused. On that exact live node, the presence of `disabled` returns a frozen `{ kind: "disabled", status: "ignored" }` result before URL resolution, root classification, Frame capture, opt-out delegation, fetch, or navigation, leaving any current request owner unchanged. Registry hosts can decode XML presence attributes with `presenceCodec`; every authored value, including the empty string and `"false"`, decodes to `true`, while absence is left to the component schema default. Turbo 8.0.23 treats `data-turbo-confirm` alone as inert during plain-link activation, so that metadata follows the same document, Frame, delegated, disabled, and request-supersession path as an otherwise identical link. Its click predicate admits `target` only when absent, empty, or exactly `_self`; those values retain the current native document, Frame, scheme, or opted-out path. Every other browser target and any present `download` attribute reject before URL resolution, fetch, navigation, or request ownership because React Native has no implicit browsing-context or download fallback. Link-owned action recognition is exact: only lowercase `advance`, `replace`, and `restore` are actions. A present blank, invalid, differently cased, or whitespace-padded value is a non-action that retains the existing document, Frame, scheme, root-delegated, or opted-out path and, like Turbo's first-value lookup, masks Frame-level action inheritance. With no link-owned attribute, ordinary top-level/delegated behavior remains unchanged, while Frame action inheritance is still pending. Turbo does not consume action metadata on scheme, root-unvisitable, cross-origin, or opted-out links, so their native browser-equivalent delegation likewise ignores it. On a visitable top-level document path, exact `advance` enters the existing advance visit and exact `replace`/`restore` reject before request ownership. A recognized action captured by a Frame passes through the shared Frame registry: an exact `advance` target promoted to the owning document works, while a Frame-local target rejects until Frame history promotion exists; promoted `replace`/`restore` reaches the document controller and rejects until document history exists. Fragment-bearing, credential-bearing, malformed, and unsupported-scheme URLs plus method/Stream metadata or the nonstandard `confirm` alias likewise reject instead of being silently reinterpreted. A link that combines `data-turbo-confirm` with method or Stream metadata still rejects because that generated-form path is not implemented. Upstream confirm metadata also suppresses hover prefetch; that behavior remains pending with native preload/prefetch support. Unsupported Turbo metadata still rejects on an opted-out node rather than relying on browser-only fallback behavior. The example registry supplies the actual `Pressable`, link accessibility role, authored-plus-pending disabled state, styling, and visible failure handling; those remain host responsibilities.

Fragment/anchor navigation and scrolling, owning-router URL/history synchronization and restoration, `replace`/`restore` document actions, Frame action inheritance and Frame-local history promotion, native download/non-current browsing-context adapters, other non-HTTP scheme policies, method/Stream-generated form links with functional confirmation, click/visit lifecycle events, snapshot restoration, and preload/prefetch remain unsupported. In Turbo 8.0.23, method and Stream links enter the form-submission path, so this hook does not silently reinterpret them as ordinary document GETs.

## Frame loading and native accessibility

A connected Frame exposes its immutable controller snapshot through a host-defined `frameComponent` boundary and `useExpoTurboFrame()`. The hook resolves the nearest connected Frame, so nested Frames receive independent bindings and components outside a Frame receive `undefined`. The default boundary remains a Fragment.

`busy` becomes true immediately while a Frame GET owns the controller and false on every terminal path. `complete` means only that the Frame is not currently loading; it is true after success, an empty `204`, failure, or cancellation. Hosts must inspect `status` and `hasBeenLoaded` when the terminal result matters. This slice covers Frame GET activity only. Document visits, targeted form submissions, delayed visual progress, submitter state, and Frame-specific live announcements remain separate contracts.

The binding includes a frozen `{ busy }` accessibility state for a host to pass to an actual native boundary. The package does not import React Native, infer that a disabled Frame makes all descendants inaccessible, announce server-authored status, or mutate `busy`, `complete`, or `aria-busy` into the logical XML tree. Controller-state changes preserve the boundary and current children; only boundary or nearest-Frame consumers rerender until a protocol mutation actually replaces children. Host boundary failures use the same registered Frame error surface as other protocol nodes.

## Programmatic Frame visits

`FrameControllerRegistry.visit(url, { frame, ... })` resolves through the shared interaction-style Frame target rules. A credential-free, fragment-free same-origin HTTP(S) URL targeting a Frame reuses that Frame's controller and loader; visiting its current `src` reloads it without applying the document root policy. An explicit action resolving to a Frame target rejects before request ownership until Frame-local history promotion exists. When the registry receives the shared document controller, a same-origin top/promoted target enters its root-aware path for the currently supported `advance` action; explicit `replace`/`restore` rejects there. The compatibility fallback directly awaits `NavigationAdapter.visit` and does not claim root-aware document ownership. A credential-free cross-origin HTTP(S) URL awaits `openExternal`. Adapter failures reject the visit without changing Frame request or source ownership.

URL admission happens before target dispatch, any host-adapter call, or any Frame request/source ownership change. Invalid, non-HTTP(S), credential-bearing, fragment-bearing, or invalid-document URLs fail as redacted `TargetError`s and invoke neither navigation nor fetch, so a rejected visit cannot cancel or replace an already-loading Frame source.

Root-visitable registered XML links now call this exact API for nearest-Frame and active named-Frame capture. Disabled presence, unsupported method/Stream metadata, non-current browser targets, and downloads reject or ignore before scheme dispatch; non-action values and browser-bypassed action metadata retain their existing path. Exact Frame-local actions reject before ownership, while exact `advance` may promote through the shared owning-document path. `data-turbo-confirm` alone is inert during activation, while the nonstandard `confirm` alias and confirmation attached to an unsupported generated-form link still reject. Authority-form `mailto:`/`tel:` URLs, empty `tel:` subscribers, fragments, raw C0/DEL controls, credentials, `javascript:`, `data:`, `file:`, `blob:`, `sms:`, and arbitrary external schemes remain rejected. XML form capture, fragment/anchor behavior, router/history state, and exact browser `Turbo.visit(url, { frame })` behavior remain separate contracts.
