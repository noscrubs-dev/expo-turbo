# Protocol

`compatibility-manifest.json` is the source-only cross-language contract for the
public TypeScript package and Rails gem. It pins the protocol and upstream
baselines, records fixture provenance, and describes each fixture's document or
Stream-fragment envelope plus its expected outcome. The additive
`behaviorFixtures` section applies shared initial-document and sibling-Stream
bytes through the public TypeScript dispatcher, then compares both the complete
ordered action report and canonical final tree with manifest-owned expectations.
Identity assertions additionally prove whether selected live nodes were retained,
replaced under the same ID, or removed.

The current `fixtures/` set proves one namespaced document, the complete
meaningful Frame envelope, a basic sibling Stream fragment, every built-in Turbo
Stream envelope (`append`, `prepend`, `replace`, `update`, `remove`, `before`,
`after`, and `refresh`), exact `target`/`targets`, morph, request-ID, scroll, and
template-root metadata, plus unsafe DOCTYPE/non-XML-whitespace Stream-text
rejections. Three executable Stream-mutation records additionally cover ordered
action isolation after an unknown action, append collision replacement, exact
`target` precedence over `targets`, per-target payload cloning, and every
built-in action's absent-template result. They also prove target retention for
`update` and selector actions, replacement for an append ID collision, and
removal for `remove` plus an empty `replace`. TypeScript and Ruby tests load the
same envelope manifest and XML bytes. Both
normalize accepted trees to the declared language-neutral AST or compact
Stream-envelope records, and both exercise every declared rejection. The Rails
helpers must also emit the exact shared Frame, basic Stream, and all-actions
records. The Frame record covers canonical identity/source, target, loading,
disabledness, autoscroll, morph refresh, recurse tokens, visit action, and
ordered application children; it is an envelope contract rather than evidence
for every Frame loading or native-device lifecycle path. Fixture readers are
test-only; neither package ships protocol sources as runtime assets.

Behavior expectations are upstream-derived from Turbo 8.0.23 and execute in
the TypeScript tree runtime. They are the first source-controlled behavioral
conformance slice. The separate native-equivalent morph record proves compatible
outer `replace`, child `update`, and selector child morphing retain the declared
target/stable-child identities while removing omitted children. This is not yet
a complete browser differential suite.

Document fixture normalization represents its sole document root; sibling Stream
fixture normalization represents its sibling Stream elements. Compact
`streamActions` assertions retain each Stream's action, optional addressing and
morph/refresh metadata, plus ordered top-level template tags. XML comments and
envelope-level XML whitespace are not part of either normalized envelope.

The package remains pre-release and does not yet claim complete Turbo compatibility.

## State references

Component-action parameters resolve state references against the action's selected document, Frame, or form scope before Zod admission:

- `{"$state":"key"}` is an exact-value reference. It must contain only the `$state` key and may resolve to any value admitted by the action schema.
- `{{state:key}}` interpolates a scalar state value into a string. Structured values are rejected instead of being coerced to `[object Object]`.
- Arrays and plain objects resolve recursively. Object keys are never interpolated.

Missing/blank keys, malformed interpolation, ambiguous `$state` objects, cycles, unsupported object prototypes, and excessive depth are typed protocol failures. Resolution never falls through from a Frame/form scope to document or app-global state. Structured exact references must come from an explicit host codec; the package does not opportunistically `JSON.parse` arbitrary XML attributes.

## Native successful form controls

`FormControlRegistry` is the host-neutral bridge between app-installed native widgets and a logical form element. A registration is bound to the exact active control node and exact active form node, not only to a reusable key or XML `id`. Without a `form` attribute, the nearest ancestor form owns the control. A present nonempty `form="id"` overrides ancestry and must resolve case-sensitively to that exact unique ID in the active tree; blank, missing, or foreign owners fail closed. The host supplies current string values and updates them as native state changes. Headless callers must unregister and re-register after changing explicit ownership. Control removal/replacement unregisters that exact control; replacing/removing the form disposes the registry. A Turbo Stream child update preserves the form registry while disposing only removed controls.

The supported control descriptors deliberately cover only caller-admitted entries: one string value, a hidden value, a checked/unchecked checkable, already-selected and enabled ordered same-name multiple values, an explicit select snapshot, a bounded `entries` snapshot, or a submitter. An `entries` descriptor must omit an outer `name` and supplies at most `MAX_FORM_CONTROL_ENTRIES_PER_CONTROL` (256) ordered `{ name, value }` pairs. Each pair has a string `name`; its value is either a string or the exact bounded `{ blob, filename }` native-file shape described below. The registry clones and freezes the entry records while retaining each admitted Blob reference, preserves pair order and duplicate names/values, leaves `_charset_` pairs literal, and places the complete list at that control's whole-document position. An empty list contributes nothing.

Control-level direct/fieldset disabledness and semantic-datalist ancestry omit an `entries` descriptor as a unit; exact form ownership, external association, validity, update, replacement, and cleanup follow the same registration rules. For single-name descriptor families, collection additionally omits an absent or exact-empty outer name, unchecked checkables, and non-selected submitters. Owned entries follow whole-document XML order regardless of registration order or whether they appear before, inside, or after the form; duplicate names, duplicate values, and empty values are preserved. A hidden control with no supplied value and a checkable with no explicit value emit `""` and `on`, respectively. The caller activates one submitter with the selection token issued by its exact live registration; fabricated, foreign-registry, stale, non-submitter, disabled, and reassigned selections fail closed. The selected submitter is appended last with an empty default value, matching Turbo 8.0.23's `FormData`-then-submitter ordering. Rejecting a disabled submitter is an intentional native safety policy because ordinary native activation cannot originate from it; this does not claim browser-programmatic `requestSubmit` parity.

A select snapshot supplies options in authored order with current selectedness, direct disabledness, and whether each option belongs to a disabled `optgroup`. Each option supplies either an explicit string value or a raw host text snapshot; an explicit value, including `""`, wins, while fallback text is stripped and collapsed using only HTML ASCII whitespace. A disabled or unnamed select emits nothing; otherwise each selected option that is neither directly disabled nor inside a disabled `optgroup` emits one entry in option order, preserving repeated and empty values. An explicit single-choice descriptor may set `defaultSelection: "first-enabled"`: when no option in the snapshot is selected, core contributes the first option that is neither directly disabled nor inside a disabled group; any selected option, even an unsuccessful disabled one, suppresses fallback. The host owns current selectedness, option order, group state, and the text snapshot; core does not discover `<select>`, `<option>`, or `<optgroup>` nodes or mutate the supplied snapshot.

An explicit hidden descriptor admits any name/value, defaults an absent value to `""`, and replaces only an ASCII-case-insensitive `_charset_` name with `UTF-8`, matching the browser hidden-input entry-list branch used by Turbo's `FormData(form)`. Authored name casing and duplicate sentinels are preserved in document order. Ordinary value descriptors named `_charset_` are not rewritten, and the manually appended submitter keeps its raw value. A value or hidden descriptor may carry one admitted directionality snapshot with a nonempty companion name and an exact host-resolved `ltr` or `rtl` value. That frozen companion follows its admitted primary value immediately, including an empty value or hidden `_charset_` replacement; disabling or omitting/emptying the primary control name suppresses both entries. For a rendered value/hidden control without an explicit snapshot, shared XML `dirname` supplies the nonblank companion name when the nearest shared `dir` context resolves to exact `ltr` or `rtl`; explicit control direction overrides inherited direction through the ordinary renderer context. If a rendered string-value control instead has absent or exact `auto` direction, the React hook scans the current live value for the first Unicode bidi class `L`, `R`, or `AL` and maps it to `ltr` or `rtl`. Matching HTML auto-directionality, a nonempty value with no strong character is `ltr`, while an empty value falls back through inherited exact direction and then `ExpoTurboProvider.defaultDirection`, which defaults to `ltr`. Explicit descriptor directionality wins. Hidden controls still require resolved XML direction. Core itself admits the resulting frozen snapshot and does not inspect locale or React Native layout state; the Expo example maps the provider default from `I18nManager.isRTL`.

Collection returns frozen semantic name/value entries, not a complete HTML entry list, query string, platform `FormData`, or physical request body. Turbo 8.0.23 constructs `FormData(form)` without the activated submitter and then manually appends only its nonempty name plus raw value or `""`; this package therefore deliberately emits neither browser image-button `.x`/`.y` coordinates nor submitter `dirname` sidecars. The `entries` descriptor is the bounded native equivalent of a FormData-valued form-associated control: the outer control name must be absent, its pairs occupy that control's tree-order slot, and no automatic `dirname` or hidden-control `_charset_` branch is applied. A native file value must contain exactly one Blob plus a nonempty filename; form admission validates Blob metadata and filename controls, and request planning admits it only for multipart with the byte/name limits below. Browser custom elements and `ElementInternals` lifecycle/state restoration remain unsupported. Automatic XML/DOM control discovery, generic picker/URI-file conversion, complete eligible-control coverage, physical accessibility evidence, and morph-aware validation preservation remain pending.

## Native constraint validation

Constraint rules are app-owned synchronous native logic, never server-selected executable code or browser APIs. A registered value, checkable, same-name multiple, select, or multi-name entry-list descriptor may carry a frozen validity snapshot: exact `{ valid: true }`, or `{ valid: false, message }` with a nonblank host-owned message. Hidden controls and submitters cannot carry validity. The registry normalizes every snapshot strictly and exposes `checkValidity()` as a pure frozen report containing every invalid control in whole-document order plus the first invalid logical key. Unnamed and externally associated controls participate because validation ownership is independent from successful-entry emission. Directly or fieldset-disabled controls, semantic-datalist descendants, hidden controls, submitters, and unconstrained controls are not candidates.

`reportValidity()` performs the same complete check and, when invalid, sends only the first invalid logical node key to the host-injected synchronous `FocusAdapter`. The focus call must return `undefined`; thrown failures, returned values, Promises, and thenables become one redacted `StateError`, submission stays blocked, and the package does not silently focus a later control. This deterministic first-invalid policy is the named native equivalent of browser validation presentation, not a claim of exact DOM focus, scroll, `invalid` event, or built-in validation-bubble behavior. The standalone Expo host maps logical keys to identity-safe `TextInput` handles, tracks user-driven focus/blur, and computes its required-field snapshot with an app-owned Zod schema.

Interactive `submit()` and `retryFailure()` validate before invoking `FormSubmissionController`. An invalid attempt resolves with `status: "invalid"`, request ID, optional submitter key, and the frozen invalid-control report; it creates no proposal, confirmation, activity/terminal revision, destination ownership, recent request-ID admission, history change, or fetch. Raw form `novalidate` presence or exact activated-submitter `formnovalidate` presence bypasses this interactive check even when authored as `"false"`. Manual `checkValidity()`/`reportValidity()` still inspect current snapshots. `successfulEntries()`, `requestPlan()`, and `submissionProposal()` remain deliberately validation-free lower-level surfaces. Server responses remain authoritative, including matching Frame/document `422` XML, and retry revalidates the current registered values before any new request.

`DocumentFormControls` is the host-owned document-lifetime collection for these registries. Repeated lookup returns the same registry while the exact form node remains active, including a child-only Stream update. Form removal or replacement disposes that identity; a same-key replacement receives a fresh registry. Explicit collection disposal releases every owned registry. React never disposes this collection. Instead, every mounted `ExpoTurboFormScope` retains the session-shared exact-form activity and defers release by one microtask; development Strict Mode replay or another provider can reacquire before release, while the last real form-scope/provider unmount cancels the exact pending confirmation or active request. Headless callers retain explicit `cancelSubmission` ownership. The provider separately reference-counts its injected document state/scope owners across effect replay and disposes them after their last real unmount.

An app-installed form component declares `formOwner: true` in its registry definition and mounts `ExpoTurboFormScope`, which also supplies the existing form state scope when configured. Descendant native components call `useExpoTurboFormControl` with their current admitted descriptor. A registered component with `form="id"` may instead call `useExpoTurboFormControl`, `useExpoTurboForm`, or both outside that form's React subtree; the exact declared owner overrides the nearest `FormContext`. The renderer subscribes above the component error boundary to the owner key, so insertion, removal, and same-ID replacement retry the association and rebind surviving controls to the fresh exact registry. A target that is missing, blank, or not declared with `formOwner: true` fails closed. Association changes form collection/submission ownership only and never moves the component into the target form's React or state-scope ancestry.

Registration follows the exact logical control node, updates in a layout effect without registration churn, unregisters on React cleanup, and binds to the resolved exact form. `useExpoTurboForm` exposes the stable form identity, frozen successful-entry collection, constraint check/report methods, exact active-form request planning, atomic destination-aware submission proposals, observable submission state/busy accessibility, frozen terminal state, and the canonical submit/cancel/dismiss/safe-retry paths. `useExpoTurboFormControl().selection()` returns the active registration-bound token passed to `successfulEntries`, `requestPlan`, or `submissionProposal` and fails after cleanup or owner rebinding; that control binding also exposes the activated submitter's pending/effective-disabled/submits-with presentation described below. Proposal re-admission rechecks that an exact external submitter still owns the captured form, so reassignment during async confirmation cannot fetch under the former owner. Rendering either hook without an active nearest or explicit declared owner fails through the registered component error boundary. Planning and proposal creation remain pure; only interactive submit/retry runs validation before delegating submission ownership to the canonical core controller.

Component definitions may declare `formContainer: "fieldset"` or `formContainer: "legend"`; the role is capability-visible and aliases resolve to the canonical definition without decoding props. Passing that component registry through `DocumentFormControls` as `formSemantics` enables native fieldset disabledness for explicitly registered controls. Core reads raw `disabled` attribute presence from every current semantic-fieldset ancestor and omits the control unless it is below that fieldset's first direct semantic-legend child. Every ancestor is evaluated independently, text/comments/non-legend siblings do not change which legend is first, nested legends are not direct legends, and explicit `form` ownership never changes the control's physical ancestry. Direct disabledness still wins. Activated submitters inherit the same rule and are rejected before planning, while `useExpoTurboFormControl` subscribes to semantic-fieldset ancestors and projects direct, inherited, and pending disabledness through its native accessibility binding. The registry does not discover controls or infer browser `<fieldset>`/`<legend>` tags automatically.

Component definitions may also declare `formContainer: "datalist"`. Every explicitly registered control physically below a semantic datalist is omitted before ordinary descriptor-specific entry construction, including value, checkable, select, bounded entries, hidden, directionality, and externally form-owned controls. Turbo 8.0.23 constructs browser `FormData` without a submitter and then manually appends the selected submitter, so this package deliberately preserves that append-last submitter behavior even when the submitter is physically below a datalist; this differs from the current HTML optional-submitter entry-list algorithm. Datalist ancestry is evaluated independently from disabled fieldsets and does not make the control disabled, so `useExpoTurboFormControl` continues to expose only direct, fieldset-inherited, and pending disabledness. The registry does not discover or infer browser `<datalist>` tags, supply suggestions, or hide datalist children automatically; the host component owns those presentation choices.

`DocumentFormControls` accepts a Turbo-compatible `formMode` of `on` (the default), `off`, or `optin`. `FormControlRegistry.shouldInterceptSubmission` and its React binding are synchronous interaction-boundary queries: `off` never intercepts; `on` requires the form and activated submitter, when present, to each have no closest `data-turbo="false"`; `optin` requires the submitter to remain navigatable and the form or any form ancestor to carry exact `data-turbo="true"`. Internal and external submitters use their own ancestry independently from the associated form, values are case-sensitive, and attributes are read live on every call. The explicit `requestPlan`, `submissionProposal`, `submit`, and retry APIs remain host-owned bypasses, matching Turbo's separation between observer interception and imperative navigation.

## Native form request planning

`buildFormRequest` is a pure boundary between collected string entries and a future form-submission controller. It accepts the owning response's final document URL, raw form and activated-submitter attributes, already-collected entries, and protocol request metadata. A named submitter must match the final collected entry exactly, preserving the registry's Turbo-compatible append-last contract without introducing a second value source. The planner resolves an explicitly present submitter action before the form action, resolves relatives against the document URL, and admits only credential-free same-origin HTTP(S). Any explicit action fragment delimiter, including an empty `#` or `?#`, fails until native navigation owns anchor behavior; it is never silently dropped.

Source method and enctype use submitter-before-form precedence. Method matching is case-insensitive for GET, POST, PUT, PATCH, and DELETE; an unsupported source method follows Turbo 8.0.23 and becomes GET. Enctype matching admits URL-encoded, multipart, and `text/plain`; an unsupported value becomes URL-encoded. Empty submitter method/enctype values fall through to the form. Attribute presence on either owner opts a GET into Turbo Stream responses, while every unsafe source method requests Streams automatically.

Input admission is staged. The planner reads each top-level option exactly once at the point it can be validated, immediately clones and freezes form, submitter, protocol, and entry scalar values, and only then reads the next field. A later getter therefore cannot rewrite an earlier compound value. Credential-free same-origin URL admission finishes before the entry-list option is read. Throwing property reads, revoked Proxies, entry-array traps, and entry-field failures become redacted `RequestError`s without retaining host causes or values. The `AbortSignal` itself remains the exact live host object by contract, but its top-level reference is captured once. Active request-plan, proposal, submit, and retry entry points stage and redact their outer option and protocol reads before they compose or dispatch through this planner.

GET replaces the action's complete existing query with the ordered entries, emits no body, ignores enctype, and does not interpret `_method`. Unsafe actions preserve their query. When Rails transport is selected, activated submitter `_method`, submitter method, first `_method` entry, then source method determine the effective method. Effective POST removes all `_method` entries; PUT/PATCH/DELETE collapse them to one lowercase override at the first occurrence or append one, and the wire method is POST. Blank, GET, and unsupported unsafe Rails overrides fail closed.

The planner admits at most `MAX_FORM_REQUEST_ENTRIES` (1,024) records and converts unpaired UTF-16 surrogates to U+FFFD before normalizing CR/LF to CRLF. It treats already-materialized `_charset_` and directionality entries as ordinary ordered records and does not rediscover their semantics; the documented `_method` normalization applies only when Rails transport is selected. URL-encoded bodies use ordered `URLSearchParams` serialization and `application/x-www-form-urlencoded;charset=UTF-8`.

String-only `text/plain` bodies implement the HTML/native-equivalent codec rather than Turbo 8.0.23's effective Fetch behavior: every normalized pair is appended without escaping as `name=value\r\n`, including a final CRLF, and an empty list produces an empty body. Empty names/values, duplicates, spaces, `%`, `+`, `=`, NUL, and other non-newline content stay literal. The exact content type is `text/plain`, and the final UTF-8 representation may not exceed `MAX_FORM_TEXT_PLAIN_BODY_BYTES` (1 MiB). GET still ignores enctype and uses its URL-encoded query path. Direct unsafe transport sends the authored method and leaves `_method` entries ordinary. Rails transport admits only an effective POST; body-based PUT/PATCH/DELETE overrides fail because Rack does not parse `text/plain` before `Rack::MethodOverride`. Hosts opting into this encoding must consume the raw body deliberately. The HTML standard itself says the format is not reliably machine-decodable because embedded CRLF and `=` are unescaped, so the gem does not install a generic Rails parameter decoder or imply that raw lines can be recovered losslessly. The standalone Rails example intentionally admits only one exact raw-body profile: UTF-8 `profile[first_name]` followed by `commit=save`, in that order with CRLF delimiters and a final CRLF. It rejects every other text/plain shape and remains a demo-host proof, not a gem-wide parameter-decoding API.

Unsafe multipart returns an immutable semantic `TurboMultipartBody` rather than encoding a boundary in core. String entries and bounded `{ blob, filename }` file entries retain document order; each Blob is capped at 10 MiB, the logical aggregate at 20 MiB, and each filename at 255 UTF-8 bytes. File entries fail every non-multipart encoding. The body deliberately has no `Content-Type`: the host `FetchAdapter` must append the entries to the platform `FormData` immediately before fetch so that platform code owns the boundary. The standalone Expo adapter implements that conversion and the Rails demo exercises it, including a system Files-picker result copied into an Expo cache-backed Blob. Core remains picker- and URI-agnostic. The returned source/effective method, encoding, entries, request, headers, and body records are cloned and frozen; protocol header metadata rejects control characters, and the planner owns no fetch, cancellation, pending UI, or response state.

`FormControlRegistry.requestPlan` composes that pure planner with the exact active form identity. It reads the form's current raw `action`, `method`, `enctype`, and `data-turbo-stream` presence; collects current registered values in XML order; and, when the caller supplies a registration-bound submitter selection, reads that node's current `formaction`, `formmethod`, `formenctype`, and Stream-marker presence while taking its name/value from the live registered descriptor. The caller owns the request ID, optional capability metadata, and abort signal; active-form callers cannot inject `frameId`. The same method is exposed by `useExpoTurboForm`. Composition is synchronous and immutable: it performs no fetch, form/Frame destination-target selection or dispatch, pending-state change, validation UI, or response handling, and a removed/replaced form or submitter fails through the existing exact-identity guards.

## Native form submission proposals

`FormControlRegistry.submissionProposal` atomically combines one exact live form/submitter snapshot with destination resolution and request planning. Logical Frame targeting uses `data-turbo-frame`; HTML `target` and submitter `formtarget` are browser browsing-context features and fail closed, including when present with a blank value. The activated submitter has precedence over the form. Inside a Frame, the first present submitter/form value then falls through to the enclosing Frame's `target` and current Frame according to Turbo 8.0.23: a blank submitter masks the form value, `_self` stays current, `_parent` selects the immediate enabled parent or document, `_top` selects the document, a missing named target stays current, and a disabled current/named/parent target selects the document. Outside a Frame, a blank submitter falls through to the form value; only an active enabled named Frame is captured, while absent, missing, disabled, `_top`, `_self`, and `_parent` targets remain document submissions.

The frozen proposal contains a public frozen destination plus the planner-issued request plan. Frame destinations derive the `Turbo-Frame` request header internally; document destinations omit it. A hidden admission record retains the issuing session, tree generation, exact form, activated submitter, exact destination Frame identities, the raw confirmation message, and the exact visit action resolved at that same snapshot. Submitter `data-turbo-confirm` wins by attribute presence; an empty value masks the form's value, whitespace is preserved, and an absent submitter value falls back to the form. Visit-action precedence starts with the same first-present submitter/form lookup, but only exact lowercase `advance`, `replace`, or `restore` is recognized. For a Frame destination only, complete absence on both submitter and form falls through to the exact destination Frame value; any present blank, invalid, differently cased, or whitespace-padded submitter/form value masks the lower-precedence source and remains history-neutral. This hidden state lets the next ownership layer reject same-key replacements and even an old tree object restored after a full document replacement without putting confirmation or history metadata into the request. The same proposal API is exposed by `useExpoTurboForm`. Proposal creation performs no fetch, cancellation, epoch acquisition, request supersession, pending UI, or response application.

## Native form transport and response disposition

`FormRequestExecutor` remains the explicit transport-only API. It owns one caller-scoped lane and returns classified immutable candidates without resolving a destination or mutating the session. `execute` first creates an `AbortController` and passes its exact signal to a synchronous `FormRequestPlanFactory`. The returned plan must retain that signal and the immutable shape produced by the planner. Only after a new plan is admitted does it abort the previous execution in that executor, so an invalid plan or thrown planning error cannot cancel valid in-flight work. Separate executor instances remain independent.

`FormSubmissionController` is the canonical destination-aware transport-and-application entry point. `submit` creates the signal first, requires the synchronous factory to return the exact session-issued `FormSubmissionProposal`, and re-admits its planner-issued plan and signal. When the hidden proposal contains confirmation metadata, a configured `FormConfirmationAdapter` receives exactly that string plus the owning abort signal and must resolve to a literal boolean. This is an intentional fail-closed native adapter contract; unlike Turbo's browser callback, other truthy values are rejected. Missing configuration fails before displacing admitted work. The valid adapter runs under a hidden exact-form owner: the public activity snapshot remains idle, the submitter remains enabled, and no destination is claimed or fetched while it is pending. Denial returns the existing frozen canceled report. Acceptance rechecks the exact form, submitter, destination Frame, and tree-generation identities before destination ownership and fetch; cancellation settles even when an adapter ignores its signal, and a late answer or rejection is inert.

Document submissions share one session-scoped lane with every `DocumentRequestLoader`; Frame submissions share the lane indexed by Frame ID while retaining the exact Frame node with every `FrameRequestLoader`. Any admitted request to the same destination supersedes its predecessor across GET and form transports. For the same exact form, `duplicateBehavior` defaults to `prevent`, including while confirmation is pending, while an explicit `supersede` admits a replacement even when it changes destinations. A confirmation-bearing explicit supersede clears the displaced request's busy/submitter presentation before awaiting the replacement answer; denial does not restore the displaced request. Different forms in different Frames and the document/Frame lanes remain concurrent. A Frame named `document` is still distinct from the document lane.

Each exact form registry publishes a stable `FormSubmissionActivitySnapshot` through `submissionState` and `subscribeSubmission`; every controller for the same `DocumentSession` observes the same exact-form activity owner. Default duplicate prevention is atomic: once a submission becomes active, another rapid submission for that exact form is prevented before it can fetch or claim a destination. The explicit `supersede` policy installs the replacement activity before aborting the incumbent, and exact-owner settlement prevents a stale abort/finalizer from clearing the newer snapshot. `cancelSubmission` cancels only the exact active owner, and form removal/replacement cancels that owner during exact-node disposal.

The active snapshot exposes busy state, request ID, and exact activated-submitter key for the immutable planner-issued request used by transport. Only that submitter receives a `FormSubmitterActivitySnapshot`: `pending` is true and `submitsWith` exposes a nonempty raw `data-turbo-submits-with` value without mutating the XML node or request. React projects these values through the form binding's `state`/busy accessibility; the control binding exposes pending/submits-with and derives effective disabled/accessibility state from authored disabled or pending. Unrelated controls and forms remain unchanged.

Activity cleanup is deterministic and precedes document, Frame, or Stream response application. Confirmation denial/failure, success, classified HTTP error, transport or parsing failure, explicit abort/supersession, form or submitter removal, and final mounted React form-scope/provider unmount all release the exact owner once; a stale confirmation or completion cannot clear or revive a newer explicit supersede. Scope ownership is keyed by the session-shared exact form rather than a registry or collection, so Strict Mode replay and multiple providers cannot cancel an incumbent prematurely. The deferred React release creates an intentional one-microtask cancellation window; headless submissions require explicit cancellation.

The same exact-form owner publishes a separate stable `FormSubmissionTerminalSnapshot` only after request activity and activated-submitter presentation have cleared. It distinguishes `applied`, `empty`, `canceled`, successful `unapplied`, uncommitted `failed`, and `committed-error` outcomes without retaining request/response bodies, field values, validation messages, confirmation text, URLs, raw host causes, or server-authored identifiers from parser/action messages. Constraint-invalid attempts never enter this terminal channel. Applied snapshots retain only the method, request ID, response classification/status, application kind, and optional submitter node key. Unapplied snapshots retain exact success classification/status plus one static reason (`visit-prevented`, `superseded`, `outside-root`, or `unvisitable-extension`), so an already-completed unsafe mutation is never mislabeled canceled or retryable when its document response does not enter the native tree. Failure snapshots use a static package-owned error name/message, allowlist only a valid numeric response status and parser location, and carry one of `safe`, `unsafe`, or `committed` retry dispositions. A valid applied `422`, `4xx`, or `5xx` XML response is therefore an applied server-owned result, never a generic client failure surface. A mutation followed by finalization failure remains `committed-error` so a host cannot accidentally present it as safely replayable.

Terminal settlement is gated by the newest admitted exact attempt. Duplicate attempts that never acquire form activity, stale superseded completions, removed/replaced forms or submitters, and late confirmation answers cannot overwrite the current snapshot or leak into a same-key replacement. Starting accepted work clears an older terminal result; a denied or otherwise canceled pre-start confirmation leaves the prior useful result visible. Exact-node disposal clears the result and invalidates recovery. React hosts may inject `formComponent` into `ExpoTurboProvider`; `ExpoTurboFormScope` renders that stable boundary around the still-mounted native fields and supplies the same binding as `useExpoTurboForm`, so result UI does not require replacing the form subtree.

`retryFailure` is explicit recovery, not automatic request replay. It is available only for an uncommitted failure marked `safe`: failure before fetch invocation or any uncommitted GET failure. A host must provide a request ID different from the failed attempt. The registry reuses the exact still-live submitter registration, recollects current control values and current form/submitter attributes, reruns confirmation and normal admission, and sends at most one newly planned request. Once a non-GET fetch was invoked the failure is `unsafe`, because the server may already have processed the mutation; a committed finalization error is `committed`. Both reject retry before fetch and require product-specific reconciliation.

An optional host `formAnnouncements` adapter consumes only the redacted non-`none` terminal snapshot plus its exact form key. React baselines the current revision on mount and claims each later revision once against the exact logical form, only after the active form has returned to idle. Initial or cleared states, pre-start denial, stale or superseded work, exact-node disposal, same-key replacement, and final-scope unmount do not announce; Strict Mode replay and another provider cannot claim the same revision twice. The package supplies no terminal user-facing copy and imports no platform accessibility API: the host maps the structured snapshot to localized text and platform delivery. Delivery failure reports through the provider error callback without changing the settled submission outcome or blocking later revisions. Constraint messages and invalid-field presentation remain control-owned and do not enter terminal announcements. Physical accessibility evidence and morph-aware validation preservation remain unsupported; transport-level request mutation and the separate form submission lifecycle are defined below.

Non-canceled response reports expose `responseUrl` as the validated final transport URL. It is evidence about the response, not an assertion that the active document URL or a Frame `src` adopted it. Applied document and Frame reports identify the actual `applicationDestination`; cross-target Frame errors can therefore report requested destination B and application destination A without ambiguity. Applied document and direct Stream reports expose `streams.interrupted`, while applied Frame reports expose `frame.streams.interrupted`; empty and canceled reports expose neither.

URL, first request-ID/header, exact Frame, proposal, plan, and signal admission all happen before a claim displaces its incumbent. Claims install the new lease before aborting old work, then recheck ownership before fetch, so reentrant abort listeners may start still-newer work without allowing the stale outer claimant to fetch. Owner-aware stale cancellation and settlement release only their exact lease and cannot clear or abort a newer owner. Ownership is scoped to the exact `DocumentSession`; Frame recurse requests retain the original target Frame lane even while an intermediary Frame ID is sent in a later request header.

The fetch adapter receives the plan's exact request object once, including its wire method, headers, body bytes/string, body content type, request ID, and abort signal. The executor retains ownership through both `fetch` and the single response-body read. Explicit cancellation or a newer admitted execution suppresses a late fetch/body resolution or rejection and returns a frozen `canceled` report instead of a network error. A current adapter or body-read failure becomes a redacted `RequestError`; canceling transport cannot prove that an unsafe request was not already processed by the server.

After the adapter follows redirects, the executor requires a nonblank credential-free same-origin HTTP(S) final URL and derives redirect metadata from either the adapter flag or a changed final URL. Raw `3xx` responses reject. A `204` is an `empty` candidate without reading a body; a blank `201` is also `empty`. Every other candidate requires either `application/vnd.expo-turbo+xml` or `text/vnd.turbo-stream.html`, buffers the body exactly once, and reports `xml` or `stream` plus `success` (`2xx`), `client-error` (`4xx`), or `server-error` (`5xx`). A Stream response is admitted only when the immutable plan negotiated it. Missing, unsolicited Stream, JSON, and HTML MIME types fail closed. `xml` deliberately does not claim whether the later dispatcher will treat the body as a document, matching Frame, or validation fragment.

The controller retains the exact destination lease through response parsing and rechecks both that lease and the hidden proposal identity immediately before the first mutation. Staged Stream dispatch also checks that the lease is still retained before each sibling action and built-in target. If synchronous notification or host action starts newer work, already committed mutations remain visible, undispatched work stops, `streams.interrupted` is true, and the newer owner remains authoritative. A `canceled` result and an ordinary `204` or blank `201` are no-ops. A negotiated Stream response dispatches once through the shared Stream dispatcher for `2xx`, `4xx`, or `5xx` without document or Frame extraction. Fatal Stream parse failures mutate nothing; per-action protocol failures retain the dispatcher's existing isolated, continue-siblings behavior, while post-commit session-finalization failures remain observable as committed outcomes.

Expo XML success applies to the frozen destination. A failed submission from one Frame to another instead transfers ownership to, and extracts, the exact origin Frame, matching Turbo 8.0.23; a top-level form targeting a Frame still uses that target because it has no enclosing origin Frame. Frame application preserves the mounted wrapper and consumes embedded Streams. Its canonical `src` changes for a successful nonempty Expo XML response, an adapter-followed Expo XML response, or a redirected `204`/blank-`201`; Stream responses and unredirected `4xx`/`5xx` or empty responses leave the existing `src` unchanged. A document response parses before replacement. Safe GETs, nonempty `201`, adapter-followed final bodies, and well-formed `4xx`/`5xx` documents are authoritative; only an unsafe nonredirect document `200` fails the Turbo redirect-safety rule. Successful document XML owns the response's validated final URL, including after an adapter-followed redirect, while every top-level `4xx`/`5xx` replacement preserves the active document URL. The adapter has already followed any redirect and supplied the final buffered body, so disposition never issues a second GET.

Every owned XML error document is applied with its `client-error` or `server-error` classification. A `422` XML Frame or document response therefore renders authoritative server values by replacement. Separately, an admitted `422` Stream may use standard `replace method="morph"`: bounded compatible-ID reconciliation retains component identity and any component-owned local state while applying the server's structural validation output. That is an explicit authoring choice, not a claim that ordinary Frame `422` responses morph or that arbitrary native input, selection, focus, or scroll state is preserved. Malformed XML, a missing matching Frame, wrong MIME, transport failure, or any stale form/submitter/destination/tree-generation identity fails or cancels without mutation. The activity contract above clears pending state before any response application and the separate terminal contract records only the newest surviving exact-form outcome.

For a document destination, optional aligned `DocumentHistory` turns the frozen action into response-driven form history without changing the request. Before confirmation or destination ownership, the controller rejects exact `restore`, rejects exact `replace` when history is absent, and otherwise requires the canonical active document URL to match the exact current history entry. Historyless default and exact `advance` submissions retain the existing history-neutral path. Once accepted document work starts, it advances the session document-navigation epoch so older promoted Frame history cannot commit afterward. A successful nonempty document response proposes from the validated final response URL: exact `replace` keeps the current index, while default/`advance` uses `proposeAdvance`, which pushes unless the final URL equals the current entry. The controller rechecks the captured history entry, navigation epoch, destination lease, and exact form proposal after fallible identifier generation. Safe GET application captures the outgoing snapshot before the history write; unsafe success clears the shared cache before application; then the acknowledged history write precedes tree replacement and embedded Stream dispatch. Authoritative `4xx`/`5xx`, direct Streams, `204`/blank-`201`, denial, cancellation, supersession, transport/MIME/parse failure, history drift, snapshot failure, and host-write failure write no form history entry. A host failure may leave a newly captured safe snapshot while preserving the old ledger/tree.

For an exact Frame destination, an exact explicit or inherited `advance`/`replace` action requires the destination's existing mounted `FrameControllerRegistry` record and configured `FrameHistoryCoordinator`; form admission never creates a controller or issues a Frame GET. Exact `restore` remains whole-document-only and fails before activity, destination ownership, or fetch; generated-form links additionally reject it before request-ID allocation. The coordinator requires fragment-free root visitability plus alignment between the active logical document URL and current history entry, and it reuses the mounted controller's stable Frame restoration scope. After exact response status/final-URL admission and before body buffering, a successful non-`204` Expo XML response optionally captures the latest outgoing whole-document snapshot when a shared cache is configured, clears an invalidating cache branch when applicable, and publishes the admitted final `src`; a potentially empty `201` delays source publication until a nonempty XML candidate exists. Successful matching XML preflights the Frame mutation, then the exact destination lane commits optional snapshot storage → Frame-scoped history → Frame children/final `src`/logical document URL. `advance` always pushes, including the same URL, while `replace` retains the current index; direct Frame visits and form promotions by the same mounted controller share one restoration identifier. Authoritative `4xx`/`5xx` XML and direct Streams publish no Frame-form history entry or logical document-URL promotion, but error XML may still replace the authoritative Frame children and Streams dispatch through the normal action path. `204`/blank-`201`, denial, cancellation, supersession, transport/MIME/parse/missing-Frame failure, history drift, snapshot failure, and host-write failure publish no Frame-form history entry or logical document/child mutation. An admitted final `src` may remain visible after later body/parse/matching failure, matching the direct Frame response path. Invalid or blank action values remain ordinary history-neutral Frame submissions.

`FormSubmissionController` optionally receives the same `DocumentSnapshotCache` instance used by document visits and promoted Frame history. Matching Turbo 8.0.23's destination-specific invalidation, an unsafe successful document XML response clears the whole cache after redirect-safety admission and before strict parsing/application; a document `4xx`/`5xx` clears after its authoritative tree replacement commits, including committed finalization failure. Document `204`/blank-`201`, wrong MIME, body/parse failure, direct Stream responses, cancellation, supersession, and network failure preserve the cache. Frame invalidation begins earlier: after exact current response status/final-URL admission but before MIME validation or body buffering, any unsafe successful or any failed non-Stream response clears the cache. This includes unsafe Frame `204`/blank-`201` plus invalid MIME, malformed XML, or a missing matching Frame on an invalidating branch. Safe successful Frame XML and every Stream MIME response preserve the cache. The cache is global to the document session, matching Turbo; the package does not pretend that clearing a separate host cache protects traversal restoration. An admitted Stream `422` follows the Stream cache branch and may use the bounded morph behavior above; it never changes ordinary XML Frame validation semantics.

## Semantic styles

Styling is component-schema-owned. A component may bind an explicit whitespace-separated `style-tokens` attribute with `tokenListCodec`; XML `class` remains structural selector metadata and is never forwarded as native `className`.

## Native direction

The shared XML `dir` attribute accepts only `ltr`, `rtl`, or `auto` when present. The renderer rejects another nonempty value, resolves the root application's value before a host document boundary, and inherits each explicit value through registered components and rendered `<turbo-frame>` boundaries. `useExpoTurboDirection()` returns the nearest admitted value, or `undefined` when no XML node established one, so the package remains React-only and leaves presentation to the host. The Expo example maps `ltr`/`rtl` to React Native `View.direction`, maps `auto`/absence to `inherit` for layout, and maps text/input direction to `writingDirection` with `auto` as its native fallback. The package does not change `flexDirection` or `textAlign`, mutate `I18nManager`, infer Unicode direction, provide a browser bidi/CSS cascade, or assign arbitrary native component direction automatically.

A host `StyleAdapter` exposes a frozen deterministic token manifest and resolves only app-compiled semantic definitions. Components create a style hook from that exact adapter with `createComponentStyleHook`, which binds the native style type and verifies the provider uses the same adapter. Resolution uses the component's canonical registered tag, including when XML uses an alias, and rejects unknown, repeated, excessive, same-group conflicting, and component-inapplicable tokens. The standalone Expo corpus demonstrates tone, spacing, layout, safe-area, and host-platform variants without accepting arbitrary React Native style objects or dynamic Tailwind/NativeWind values.

Style composition is deterministic: component defaults are lowest priority, admitted independent token groups compose in authored order, explicit schema-owned semantic props override tokens, and local transient native state such as pressed/disabled remains the component's highest-priority layer. Unsupported input raises a typed props failure inside the registered component error boundary; it is never ignored or normalized into a fallback class.

## Mutation identity and cleanup

Stable XML IDs provide addressability; they do not make two different tree nodes or document sessions the same runtime instance. A plain Turbo Stream `update` preserves its target node, component state, Frame controller, and target-owned scope while replacing children. The bounded exact `replace target="…" method="morph"` contract likewise preserves its compatible ordinary application-element target; plain `replace` and append/prepend/before/after ID-collision replacement create a new node identity even when the authored ID and tag are reused. React remounts that instance, old disposal hooks run exactly once, and a same-ID Frame wrapper cancels its old request/controller before the replacement can connect. Frame requests also carry exact node and controller ownership, so a retained stale controller can neither cancel a replacement request nor commit a late response into the replacement wrapper.

Turbo 8.0.23 selects structural morphing only when a `replace` or `update` Stream carries the exact lowercase `method="morph"`. The native dispatcher supports `update` when `target` or `targets` resolves ordinary application elements or Frames, and supports `replace` when the shared template contains exactly one ordinary application-element root whose exact tag, local name, namespace, and prefix match each resolved target. The retained outer replacement root may change or remove its ID; a new ID already owned by an active descendant fails the isolated action. In either case it builds and validates the entire child plan before one session mutation. An ordinary application element with a stable ID retains identity when its exact shape agrees even if the incoming tree moves it to another ordinary parent inside the same morph root. Newly introduced or incompatible ordinary wrappers are planned recursively and can receive such a retained descendant. For same-parent ID-less ordinary application wrappers with the same exact QName, planning builds Idiomorph-style sets from the unique IDs that persist across the current and incoming morph roots. The first current wrapper whose set intersects the incoming wrapper's set wins; only a current wrapper with no persistent descendant IDs may serve as the deterministic soft fallback. When an incoming state-free wrapper precedes two further soft-compatible siblings and at least two current candidates remain, the first wrapper is inserted instead of consuming current state, matching Idiomorph's bounded prepend lookahead. Together these rules preserve reordered state-bearing wrappers and prevent a newly prepended wrapper from consuming later sibling state. An admitted outer `replace` retains the matching root as well, updating its attributes and recursively ordered children; incompatible and protocol-wrapper descendants structurally replace, while compatible stable descendants of an incompatible ordinary wrapper remain retained. A current ordinary application element carrying `data-turbo-permanent` is admitted only with a nonblank stable ID. One compatible incoming identity anywhere in the morph root moves that current logical node into the incoming position while retaining its attributes, complete subtree, document/form/external-state ownership, and zero morph revision; the incoming marker itself may be absent. A directly omitted current permanent remains in its old sibling position. An incoming-only marker is reconciled or inserted normally and becomes opaque for subsequent morphs. An outer retained permanent owns nested permanent descendants without independently matching them. Under the same rendered parent, a retained registered component also preserves its mounted React state by default. Its capability-visible definition may instead declare `morphState: "reset"`; after a successful Stream, direct Frame reload, or current-document morph commit, only that exact retained component boundary remounts and runs ordinary resource/disposal cleanup while the logical XML node remains retained. The policy is driven by a per-node morph revision and therefore does not fire for ordinary non-morph mutations. When the optional `AutofocusAdapter.getFocusedId()` reports that exact stable node key before the reset commit, the persistent React boundary checks focus again after the replacement component's layout effects. It restores only when focus became empty and `canFocus()` still admits the same key; focus that moved to another key wins. A successful focus restoration then offers the exact key to the optional `AutofocusScrollAdapter`, using the same availability and synchronous request contract as document, Frame, and Stream autofocus. Snapshot, availability, focus, and reveal adapter faults become redacted state errors through the registered component boundary. A duplicate/out-of-subtree ID, an ID-bearing multi-target payload, a Frame replacement target, an invalid outer replacement root for an existing exact target, protocol reparenting, a protocol/target/ancestor/outer-root permanent marker, a marker-bearing application node without a stable ID, an incompatible same-ID permanent identity, or omission of an unmatched ordinary ancestor containing a permanent descendant fails the one action without changing tree, revision, snapshots, scopes, or disposal registrations; later sibling Streams still run. The separate bounded direct Frame reload and current-document Refresh Stream morphs are documented below. Successful controller-owned current-document morphs also emit the page-level logical `morph` notification documented below. Unsupported morph shapes, DOM targets/bubbling, and Idiomorph's DOM-active-element cursor heuristic remain outside the contract. Missing, blank, differently cased, and arbitrary method values retain ordinary plain `replace`/`update` behavior; they are not rejected or normalized into morphing.

Mounted React state preservation has one native ancestry boundary. A retained compatible node under the same rendered parent—including keyed sibling reorder and matched anonymous-wrapper reorder—keeps its component instance unless its definition selects `morphState: "reset"`. A retained logical node moved to a different ordinary parent keeps its tree object, stable key, and document/form/external-state ownership, but React Native remounts the registered component because its React parent changed. The old component unmounts and its `useNodeDisposal` callback runs before the new instance registers fresh component-owned resources; the logical node itself remains active. This is the native observable contract; it does not claim browser DOM node reparenting or uncontrolled-input selection.

The optional `AutofocusAdapter.getMorphFocusedId()` snapshot closes the native remount focus gap without changing standalone Stream `getFocusedId()` polling. The document renderer captures the stable focused key immediately before a package-owned native morph transaction. It stages restoration only when that key still resolves after commit and its morph revision advanced, whether reconciliation retained the exact logical node or replaced an incompatible application node under the same stable ID. After descendant layout effects register the new handle, focus is restored only when focus became empty and the exact node/revision still owns the key; focus that moved elsewhere during commit wins. A successful restoration may use the optional autofocus-scroll adapter to reveal that key. Missing adapters, unavailable or removed IDs, ordinary non-morph mutations, and whole-tree replacement are no-ops. This is logical native focus continuity, not browser selection-range or uncontrolled-input cursor preservation.

Built-in compound Stream actions validate every payload ID, including repeated IDs within one template, before removing collision nodes. A rejected action therefore leaves the active tree, revision, snapshots, scopes, and disposal registrations unchanged while later sibling Stream actions may continue. This preflight guarantee applies to the built-in dispatcher; arbitrary host mutations and custom Stream handlers must not mutate and then throw as though they were transactional.

Registered component cleanup runs on logical subtree removal/replacement and on ordinary React/provider unmount. Cleanup is identity-bound and once-only, so a logical mutation followed by React reconciliation cannot dispose the same resource twice.

## Document GET transport

`DocumentRequestLoader` is a host-neutral transport and tree-replacement primitive for a mounted `DocumentSession`; the observable lifecycle controller is layered on it below. It is not a navigation, history, or cache controller. It resolves relative sources against the active document URL and permits only credential-free, same-origin HTTP(S) request and final-response URLs. Document GETs advertise the Expo Turbo MIME type and send protocol, runtime, request-ID, and optional capability headers without a `Turbo-Frame` header.

After the fetch adapter follows redirects, the response's final URL becomes the parsed document URL. Valid XML responses are classified as `success` (`2xx`), `client-error` (`4xx`), or `server-error` (`5xx`) and replace the active tree only after MIME validation and a complete parse. A valid `4xx` or `5xx` XML document still commits as authoritative server output while retaining its distinct classification for the visit controller. An optional synchronous `beforeCommit` admission callback receives only frozen response metadata, candidate status, and the candidate document's normalized root location; it can commit or discard the candidate but cannot access or mutate the private parsed tree. An empty candidate reports the active document's root.

A `204` response and a blank `201` response are deliberate native empty outcomes that preserve the current tree and URL; they are not a claim of exact browser Drive GET behavior. Raw `3xx` responses, missing or cross-origin final URLs, wrong MIME types, malformed XML, and transport failures reject with typed errors without replacing the current tree.

Explicit cancellation, supersession by a newer document GET or document-destination form request from any loader/controller instance, or any intervening session-tree replacement prevents the stale response from committing and produces a canceled outcome. Tree ownership uses a monotonic generation, so restoring an earlier tree object does not revive an earlier request. `cancel(owner)` returns true only when that exact active request was canceled; stale/not-owned cancellation and protected final-commit cancellation return false. Stale owner cancellation cannot cancel a form request that has since claimed the document lane. Disposal failures raised after tree replacement remain visible even though the new tree already owns the session.

An optional synchronous `onRequestStart` callback is the exact post-claim acknowledgement boundary. It runs only after the document lease is installed and still owned, before recent-request bookkeeping or fetch, and must return `undefined`; a value or thenable is rejected before transport. The loader rechecks ownership after the callback, so synchronous supersession cannot continue into request bookkeeping. A blocked or already-superseded claim never invokes the callback.

The raw loader does not decide navigation actions, root visitability, visit/render lifecycle events, snapshot/history restoration, preload/prefetch policy, or form submission. Its optional shared request lifecycle is transport-only; the separate higher-level lifecycles below own click, `frame-missing`, Stream behavior, controlled native `before-frame-render`, logical `before-frame-morph`, renderer-acknowledged document and mounted-Frame render/load, narrow native document reload, and the bounded direct Frame reload morph described below.

## Request lifecycle

A host may inject one `RequestLifecycle` into `DocumentRequestLoader`, `DocumentPreloader`, `FrameRequestLoader`, `FormRequestExecutor`, or `FormSubmissionController`. Each admitted transport operation takes a stable listener snapshot and dispatches independently, so pausing one request never serializes unrelated document, Frame, or form work. Frozen context distinguishes document load from preload, identifies the outer/requested Frame and recurse depth, or includes the exact form node when available.

`before-fetch-request` is cancellable and pausable. Its `RequestMutation` exposes cloned URL, method, headers, optional body, and the original live `AbortSignal`. After listeners settle, the package clones and re-admits the result against the calling transport's URL, allowed-method, body-size/encoding, negotiation, cache-key, and protected-header policy; a rejected mutation never reaches `FetchAdapter`. Document and Frame traffic remains bodyless `GET`, document preload cannot retarget its reserved cache key, and form mutation remains limited to same-origin, fragment-free admitted requests and supported string bodies.

`before-fetch-response` receives the admitted request and frozen response metadata. Prevention returns an explicit `prevented` result before body reading, parsing, cache writes, or document/Frame/Stream application; unsafe prevented form responses still follow cache-invalidation safety. A non-abort adapter rejection dispatches cancellable `fetch-request-error` with a package-owned redacted `RequestError` and then still rejects transport. Preventing that event suppresses the package's default visit/Frame/form error recovery and observer reporting plus refresh, traversal, and automatic-preload error reporting; it never relabels the operation as canceled, prevented, or successful, and ownership, busy presentation, and finalization cleanup still run. The host that prevents the event owns any replacement recovery/reporting. Abort or supersession dispatches no error event and settles as canceled even when an adapter or listener ignores abort. This is request lifecycle only; the separate higher-level lifecycles below own click, `frame-missing`, Stream behavior, controlled native `before-frame-render`, logical `before-frame-morph`, renderer-acknowledged document and mounted-Frame render/load, and narrow native document reload.

Form terminal reports distinguish the admitted wire `transportMethod`. If a safely retryable failure already invoked a GET adapter, later live replanning and lifecycle mutation must remain GET; an unsafe retry is rejected before fetch and the constraint survives. An unsafe request that failed before adapter invocation retains the existing safe retry path.

## Form submission lifecycle

A host may inject one `FormSubmissionLifecycle` through `FormSubmissionControllerOptions.submissionLifecycle`. Registered native forms, their explicit retries, and Turbo generated-form links share that controller-level instance. `submit-start` and `submit-end` are synchronous notification-only native events with a stable listener snapshot. Every listener must return exactly `undefined`; throws, scalars, Promises, and thenables become redacted observer faults, later observers still run, and rejected thenables are consumed. One optional `onObserverError` receives the resulting `AggregateError`; if it is absent or fails, the redacted aggregate surfaces on a microtask without changing transport, pending state, or tree truth.

After confirmation and a fully admitted/resumed `before-fetch-request`, the controller completes native history/navigation admission, publishes exact form busy state plus activated-submitter pending/disabled and `data-turbo-submits-with` presentation, and emits `submit-start` immediately before recent-request-ID admission and physical fetch. Its frozen detail contains one identity-bound `formSubmission` handle: a cloned destination, exact form/source-link node key, optional submitter node key, request ID, live owning signal, `waiting`/`stopping`/`stopped` state, and `stop()`. It deliberately exposes no URL, headers, body, field values, confirmation copy, raw response, or host error. The first `stop()` while waiting returns `true`, moves the handle synchronously to `stopping`, and aborts only that exact owner. Remaining start observers still run while busy/submits-with presentation remains active; synchronous cancellation never invokes the adapter, request-finished cleanup then emits one end, and later `stop()` calls return `undefined` without effect.

`submit-end` fires exactly once for every emitted start when an admitted response reaches the request-finished boundary—after `before-fetch-response`, before response-body buffering—or when a pre-response failure/abort settles. Exact busy/submitter presentation is cleared before observers run. End therefore precedes content-type/body admission, strict XML parsing, history/tree mutation, Stream dispatch, owning-document visit events, promoted-Frame finalization, autofocus, and terminal-result settlement. The same frozen handle is now `stopped`. Its frozen detail mirrors Turbo's top-level shape: cancellation or a default-handling-suppressed fetch error adds no outcome fields; a received response adds response-derived `success` plus only frozen status/redirect metadata under `fetchResponse`; and an unsuppressed pre-response failure adds `success: false` plus a package-owned frozen, redacted `RequestError`. Received `2xx` responses report success, received/prevented `4xx` and `5xx` report failure, and an admitted response remains authoritative if body buffering is later aborted, returns the wrong MIME type, or fails during parse/application. Application/terminal truth remains a later separate contract.

Constraint-validation failure, nonintercepted submission, duplicate prevention, confirmation denial/failure, planning/history/ownership rejection, and canceled or failed `before-fetch-request` admission emit neither event. Abort, exact form/submitter removal, provider-scope cancellation, destination supersession, and start-listener reentrancy after an emitted start still produce one owner-safe end; an older end cannot clear, settle, or apply over newer work. React Native has no DOM event target, bubbling/composed path, `HTMLFormElement`, browser `FetchResponse`, or browser form-submission mutation object. Request mutation remains exclusively in the preceding `before-fetch-request` phase.

## Stream render lifecycle

A host may inject one `StreamLifecycle` through direct `StreamActionDispatchOptions.streamLifecycle`, `FrameRequestLoaderOptions`, `FormSubmissionControllerOptions`, or `CableStreamSourceRegistryOptions.streamOptions`. Every source then shares the same ordered dispatcher: direct fragments, embedded document or Frame Streams, negotiated form responses, and Cable messages. The standalone Expo runtime owns one shared instance for its direct controls, Frame and form controllers, and refresh action.

For each admitted `<turbo-stream>`, cancellable `before-stream-render` fires synchronously before action validation, target/selector resolution, or mutation. Its frozen detail exposes the exact parsed `newStream`, normalized action string, fragment index, and a controlled mutable `render` function. Prevention is irreversible, consumes only that Stream, returns a frozen `status: "canceled"` report, emits the action notification below, and continues later siblings; it is distinct from a valid stale target's `noop`. A pre-mutation throwing listener, non-`undefined` return, invalid replacement, or synchronous renderer failure becomes one package-owned redacted action error without preventing later siblings. A package-issued committed failure remains fatal because logical tree truth has already changed and finalization failed; host-constructed lookalike errors are not trusted. An asynchronous callback rejected after mutating the session also stops dispatch as redacted committed truth rather than letting later siblings run over an inaccurately reported state.

The default renderer may run at most once through the exact frozen `StreamRenderContext`, and its continuation expires when the registered renderer settles. A wrapper may await that continuation but must return its exact result; a complete registered replacement may instead resolve to an admitted `{ status, matchedTargets, appliedTargets }` result. Counts must be nonnegative integers, `appliedTargets` cannot exceed `matchedTargets`, and `noop` applies zero targets. The dispatcher awaits native Promise/thenable renderers before the action notification and next sibling. It invokes captured `then` methods without package-added delay and consumes bounded nested resolution without allowing self-resolution to starve the runtime. An optional host-owned `streamRenderScheduler.beforeRender()` is also awaited after `before-stream-render` admission and before the selected renderer. Frame loaders and form-submission controllers snapshot and propagate that scheduler to embedded Frame Streams and negotiated Stream responses instead of silently dropping it. This provides Turbo's logical awaited-render and pre-render scheduling points without claiming that an arbitrary host scheduler produced a physical browser paint. Registered native callbacks are trusted host code and may use the live session; the package detects failures around mutations it observes, but cannot roll back arbitrary host side effects scheduled after the callback settles. Server XML can never install executable renderers.

After a standalone direct fragment, Cable message, or negotiated Turbo Stream form response commits a default-rendered exact-target `replace`/`update method="morph"`, native-only `stream-morph` fires synchronously with frozen `{ action: "replace" | "update", index, targetId }` before the final action notification below. It is emitted after the bounded tree mutation, so listeners observe committed logical tree truth; a renderer wrapper that calls `renderDefault()` remains eligible, while a complete registered replacement does not. Plain methods, selector-only/missing targets, canceled/no-op/error actions, and Streams embedded in a document or Frame response emit no `stream-morph`. This is a narrow action-level native observation contract, not Turbo's per-retained-DOM-node `turbo:morph-element`, an attribute event, or a physical renderer/repaint boundary.

After the final report is known, native-only `stream-action` fires synchronously before the next sibling. Its frozen detail contains the exact Stream node and a frozen cloned report with `applied`, `noop`, `canceled`, or `error` status; exposed errors retain only package-owned action context. Turbo 8.0.23 has no corresponding after-action DOM event, so this notification is an explicit native observability contract rather than an Exact browser claim. Listener snapshots are stable, and a listener's thrown or returned fault cannot rewrite the completed report: one optional lifecycle `onObserverError` receives the redacted `AggregateError`, while an absent or failing observer surfaces that aggregate on a microtask. These listeners remain trusted synchronous host code and may deliberately mutate the session or supersede ownership before a later sibling, so the package does not claim to isolate arbitrary observer side effects. A concrete React/native host still owns its scheduler implementation and physical commit evidence; broader morph lifecycle integration remains separate.

## Semantic-link click lifecycle

The `DocumentVisitLifecycle` injected into the exact `DocumentVisitController` also owns the native semantic-link `click` event used by `useExpoTurboDocumentLink`. React reaches that lifecycle only through the controller's internal dispatch boundary, so a link cannot accidentally emit through a different lifecycle than the ensuing document visit. The synchronous cancellable event has frozen `{ nodeKey, url }` detail: `nodeKey` identifies the exact live registered component and `url` is the normalized fragment-free destination, or the canonical same-document fragment URL for the bounded anchor path below. Listener snapshots are stable, only `undefined` is accepted, and a thrown, scalar, Promise, or thenable result fails through a package-owned redacted error before routing begins. The event and its cancellation state are frozen against listener-side property replacement.

For ordinary non-anchor activation, one event fires only for an enabled, supported, root-visitable plain semantic link after exact node identity, URL, target/download, root, and `data-turbo` admission, then before nearest/named Frame capture or top-level `before-visit`. The canonical destination is captured before listener dispatch, so a listener-side document URL change cannot re-resolve a relative link. Prevention returns frozen `{ kind: "link", status: "canceled", url }` evidence and allocates no Frame/document request, request ID, history proposal, form activity, or navigation work. The bounded same-document anchor path is the explicit exception: it emits its cancellable `click` after its narrower admission and before revalidation, without root-policy or visit ownership. React Native has no browser default navigation to resume and exposes no fabricated `MouseEvent`, bubbling/composed path, modifier-key state, or DOM target. Generated method/Stream links are intercepted by the form-link path before click; ordinary forms have only form/request lifecycle; opt-outs, cross-origin/outside-root/excluded-extension delegation, `mailto:`/`tel:`, disabled links, host-delegated target/download metadata, and non-anchor stale activations emit no click. Interactive root-unvisitable HTTP(S) links delegate directly to the host adapter and therefore emit no link-driven `before-visit`; a host's explicit programmatic `DocumentVisitController.visit` retains its own existing `before-visit` contract.

## Owning-document visit lifecycle

A host may inject one `DocumentVisitLifecycle` through `DocumentVisitControllerOptions.visitLifecycle`, `FormSubmissionControllerOptions.visitLifecycle`, or both when those controllers share a document session. The lifecycle is synchronous and takes a stable listener snapshot per dispatch; listeners must return exactly `undefined`. A thrown listener, returned scalar, Promise, or thenable fails through a package-owned redacted error, and returned thenable rejection is consumed. `visit`, `before-cache`, `morph`, `render`, `load`, and `reload` notification faults never change transport, cache, or tree outcomes: the lifecycle continues later observers and offers one redacted `AggregateError` to its optional `onObserverError`; without a successful observer, that redacted aggregate surfaces on a microtask.

Every controller-owned network document GET that reaches an admitted parsed tree—including a reused prefetch response, canonical preview revalidation, current-document refresh, and network history restoration—every cached preview/final snapshot application, and every successful document-form XML response dispatches `before-render` after successful morph, cache-clone, or strict form-tree preflight and before outgoing snapshot/history commitment, render-ticket creation, embedded Stream dispatch, or logical tree mutation. Its frozen detail exposes read-only `{ currentDocument, newDocument, renderMethod: "replace" | "morph", url, render }`; the pausable/cancellable event itself remains live so a synchronous listener can call `pause()`/`resume()`, call `preventDefault()`, or replace `render` with an asynchronous wrapper around the controlled one-shot `renderDefault()` continuation. The package retains exact request, snapshot, or form destination ownership while waiting. Prevention, explicit cancellation, or supersession leaves the active document unchanged; a missing default delegation, duplicate/late delegation, non-`undefined` result, or listener/renderer failure rejects redacted before package mutation. Successful form prevention produces `render-prevented` unapplied terminal truth after request and visit lifecycle cleanup. The wrapper admits only the already-validated package transaction and cannot substitute an arbitrary server-selected native tree.

After a successful controller-owned bounded current-document morph transaction, notification-only `morph` fires synchronously before any renderer acknowledgement. Its frozen detail is `{ currentDocument, generation, newDocument, url }`: `currentDocument` is the retained active logical document after commit, `newDocument` is the admitted parsed source document, `generation` is the new authoritative session generation, and `url` is the final response URL. Listeners therefore observe committed logical truth in both rendered and headless hosts. Ordinary replacement and authoritative error documents, failed morph preflight, empty/canceled/superseded work, direct Frame morphs, and Stream morphs emit no page event. This is the native equivalent of Turbo 8.0.23's post-reconciliation `turbo:morph`; it deliberately provides logical handles instead of DOM targets/bubbling and makes no physical-commit or repaint claim.

`before-visit` contains the frozen canonical `{ url }` proposal and is cancellable. `visit(source)` dispatches it after action, URL, root, and required history alignment admission but before history-identifier generation, cache lookup, request-ID allocation, destination ownership, state publication, fetch, or host navigation. Accepted current-document refresh dispatches the same event after current/idle URL alignment and before its optional history proposal. Prevention returns frozen `{ source: "visit-lifecycle", status: "canceled", url }` evidence without publishing a controller state transition or disturbing incumbent work. A listener-started newer proposal wins through the controller's proposal epoch. Root-external and excluded-extension `advance`/`replace` proposals receive `before-visit` before host delegation but no `visit`; invalid proposals and direct host paths that bypass the controller emit neither event.

`visit` contains frozen `{ url, action, direction }` detail and is notification-only. Direction defaults to Turbo 8.0.23's `forward` for advance, `none` for replace, and `back` for restore; an explicit admitted direction is retained across preview/revalidation and restore cache-miss fallback. Its URL is the canonical logical document proposal; an admitted `before-fetch-request` URL mutation may retarget the physical request without rewriting that visit identity. Network visits publish `started` after exact document ownership, invoke any installed `before-fetch-request` listener phase, then emit `visit` before pause/resume or package-specific cancellation settles, navigation bookkeeping, recent-request-ID admission, and physical fetch. A canceled pre-fetch request therefore still reports the logical visit that started, while no physical request is fabricated. A cached preview emits once after preview ownership and `started`, before provisional replacement; its canonical revalidation emits no second event. Explicit cached restore and accepted current-document refresh report `restore` and `replace` respectively. Host-driven traversal intentionally skips `before-visit` because native history has already moved; both cached and network traversal emit one `visit` with `action: "restore"` and the direction derived when that host entry was adopted.

After a successful form request has finished transport and cleared its exact form/submitter activity, an injected form-controller lifecycle receives one final-response-URL `before-visit`. The activated submitter/form's exact `advance` or `replace` action wins; otherwise only a redirect back to the canonical active URL defaults to `replace`, while every other successful document response uses `advance`. The response body has already been read and an unsafe successful request has already invalidated snapshots, but parsing, history, and tree replacement have not started. Prevention or lifecycle-listener supersession leaves history/tree unchanged and returns a frozen `status: "unapplied"` report plus terminal snapshot with `classification: "success"` and reason `visit-prevented` or `superseded`; unsafe cache invalidation is intentionally not rolled back.

An admitted proposal is then classified against the active document's Turbo root. A root-visitable final URL emits one `visit` and proceeds into strict parsing and guarded history/tree application. An outside-root or excluded-extension final URL requires `FormSubmissionControllerOptions.navigation`, awaits exactly one `navigation.visit(finalUrl, action)`, emits no `visit`, skips parsing/history/tree application, and returns successful `unapplied` truth with the matching static reason. Same-origin response admission makes external final URLs impossible. A missing adapter fails loudly, and an asynchronous navigation rejection is awaited and redacted through normal form failure state rather than publishing false unapplied success. Because Turbo proposes a Visit before rendering, malformed successful document XML retains the already-emitted `before-visit` → `visit` pair before strict parsing fails. Authoritative `4xx`/`5xx` documents, direct Streams, empty responses, Frame responses, prevented response handling, transport/MIME failure, cancellation, and stale pre-lifecycle work emit no form-driven owning-document visit.

After a successful root-visitable document GET or refresh follows a redirect and commits its final history/tree, the lifecycle emits Turbo's second `visit` with the final URL and `action: "replace"`. It emits no second `before-visit`, performs no second fetch or controller-state start, and preview revalidation produces only the original requested event plus this final follow-up. The follow-up observer sees committed truth while the original controller lifecycle is still `started`; if it starts newer work, the older settlement cannot finish that newer lifecycle. Redirected error documents, empty responses, prevented/canceled/stale work, host traversal, and redirects discarded for host delegation emit no follow-up.

An exact local `advance`/`replace` promotion shares this lifecycle through its `FrameHistoryCoordinator`. Direct, inherited, programmatic, and lazy Frame GETs plus successful matching Frame-form XML emit the final admitted response URL only after Frame-scoped history, Frame children/final `src`, logical document URL, embedded Streams, controller completion or form-activity cleanup, and autofocus intent are logically committed. When the exact mounted Frame has a retained renderer, React acknowledgement, autoscroll/autofocus, and the ordinary `frame-render`/`frame-load` notifications below also precede the synthetic document lifecycle; headless promotion keeps the logical post-commit path without inventing visual events. Matching direct-GET `2xx`/`4xx`/`5xx` XML may promote; failed Frame forms, ordinary Frame loads, empty/prevented/canceled/stale or structurally rejected responses, `_top`, and external paths emit neither document event. Late `before-visit` prevention suppresses only the paired synthetic visit and never rolls back committed Frame/history truth. A listener-started newer document or destination owner similarly suppresses the stale notification. A redirected promotion emits exactly one pair for the final URL and original action, never the document redirect's second follow-up. Root-visitable finals emit `visit`; outside-root or excluded-extension finals await one required `FrameHistoryCoordinatorOptions.navigation.visit` and emit no `visit`. Missing or rejected navigation becomes committed-error truth.

Final promoted-Frame classification uses the absolute Turbo root captured at admission, so a relative root cannot move when the logical document URL changes. A committed embedded-Stream or autofocus finalization failure emits neither lifecycle event. While required host navigation is pending, newer document or exact destination ownership and mounted Frame release cancel only the stale wait, consume a late adapter rejection, and preserve the already-committed Frame/history truth.

A mounted `ExpoTurboRoot` turns controller/form whole-document replacements into renderer-acknowledged lifecycle work only when the same `DocumentVisitLifecycle` is injected. The package records the next tree generation before replacement and seals the ticket only after response-owned logical mutations finish. React acknowledges only the same live document and generation at its exact current session revision, after the sealed revision; observer-induced revision changes require a fresh commit but never duplicate `render`. Frozen notification-only `render` detail is `{ generation, preview, renderMethod: "replace" | "morph", url }`; only a successful bounded current-document Refresh Stream morph reports `"morph"`. It fires in the root layout phase after the registered React tree or root `renderError` fallback commits and before any applicable document autofocus. Strict Mode replay, multiple mounted roots, a superseding generation, or final renderer release cannot duplicate or strand the ticket. An explicit controller cancellation after logical document installation releases only its pending visual wait: the installed response remains completed/failed while `render`, `load`, and response autofocus stay suppressed. Initial projection, headless work, direct session or standalone Stream mutation, Frame application, and preload create no render ticket.

`DocumentVisitController` remains `started` until the prepared render is acknowledged. A cached preview emits `render`, suppresses autofocus and `load`, and must acknowledge before canonical revalidation starts. An authoritative committed cache, network, refresh, or traversal document emits `render`; after any applicable autofocus, the controller publishes `completed` for success or `failed` for committed `4xx`/`5xx`, then emits frozen `load` detail `{ generation, url }`. Empty `201`/`204`, prevented, canceled, discarded, uncommitted, unavailable, or failed render work emits no `load`.

A document-targeting `FormSubmissionController` creates the same ticket immediately before replacement and seals it only after embedded response Streams finish, preventing an intermediate tree revision from acknowledging early. Successful and authoritative `4xx`/`5xx` document XML emit `render`; only successful classification emits `load`, after form activity cleanup and terminal publication. Direct Streams, Frame responses, empty, prevented, canceled, and unapplied work emit neither visual event. These are native equivalents: they expose no DOM target or `newBody`, guarantee no physical paint/`nextRepaint()` timing, and expose no browser load timing metrics.

All Frame morph other than the bounded direct reload-morph case below, morph element/attribute events, and browser DOM/physical-paint timing parity remain pending. Automatic visit-control response promotion and `frame-missing` remain the separate response paths documented below.

## Document snapshot cache foundation

`DocumentTree.clone()` is the production deep-copy boundary for history and cache work. It preserves the document URL by default, or may retarget only the clone to a supplied nonblank restoration URL, while retaining exact deterministic/internal keys, node kinds, ordered children, attributes, namespace metadata, source locations, indexes, and the source tree's next safe mutation-key sequence with independent document, node, attribute, location, parent, and child records. Mutating either tree cannot affect the other. The optional temporary-element policy omits each subtree whose root has `data-turbo-temporary` by attribute presence or deprecated `data-turbo-cache` with the exact value `"false"`. Other `data-turbo-cache` values, including `"False"`, empty, and whitespace-padded values, remain in the clone; the policy never removes a subtree from the live source tree or affects active rendering.

`DocumentSnapshotCache` defaults to Turbo 8.0.23's ten-entry LRU. It accepts normalized credential-free absolute HTTP(S) locations, removes the fragment from the cache key, and retains path plus query identity. A write requires the tree's own absolute document URL to match that normalized key, preventing a restored tree from resolving relative links, forms, or Frames against an unrelated base. `has` does not affect recency; `get`, `getPreview`, and `put` move an entry to the newest position. `put` stores a temporary-pruned clone, and every successful retrieval returns another fresh clone, so a restored or previewed document cannot mutate the retained snapshot or the document that produced it. Replacing a key updates its bytes and recency; capacity eviction removes the least recently touched entry; `size` and `clear` expose bounded host-neutral management.

The sole XML root may declare the native cache-policy equivalent `data-turbo-cache-control`. Turbo's two flags are independent: exact lowercase `no-cache` means `{ cacheable: false, previewable: true }`; exact lowercase `no-preview` means `{ cacheable: true, previewable: false }`; every other or nested value uses `{ cacheable: true, previewable: true }`. All returned policy records are frozen. Cache admission skips a `no-cache` tree before cloning or changing LRU state, so an older matching entry remains available. Callers use `get` for unconditional restoration retrieval and `getPreview` for preview-eligible retrieval. `getPreview` performs the same LRU read first, then suppresses a retained `no-preview` clone, matching Turbo's lookup-before-previewability ordering without deleting the restoration snapshot.

`DocumentPreloader` is the bounded host-invoked full-document preload foundation. `preload(source)` first resolves the explicit string source against the active document and requires the current root policy to classify the credential-free same-origin HTTP(S) result as visitable; fragments fail closed because no anchor preload contract exists. It reserves that canonical URL and exact pending Promise before invoking the shared cache, request-ID adapter, or transport, so reentrant equivalent calls share one operation and cancellation can stop work before either an ID or request exists. An existing cache entry then returns a frozen `hit` without allocating a request ID. A miss sends one bodyless GET with the standard document protocol/version/request-ID/capability headers plus Turbo 8.0.23's `X-Sec-Purpose: prefetch`, while deliberately publishing no document lease, navigation epoch, recent request ID, visit lifecycle, history, progress, or active-tree mutation.

Only a `2xx` response with a validated same-origin final URL, whose MIME is `application/vnd.expo-turbo+xml`, and whose body is nonempty strict XML can reach the shared snapshot cache. A changed or explicitly redirected same-origin final URL remains cache-only: the original requested URL stays the parsed-tree base, cache key, and report URL, while the final URL receives no alias, navigation/history write, or active-tree mutation. Configuration and response metadata are snapshotted once; every configured parser limit must be a positive safe integer before requests begin, and the root cache policy applies. A cacheable tree is cloned and temporary-pruned by `DocumentSnapshotCache`; a root `no-cache` document returns the distinct frozen `not-cacheable` report and is not retained. For a cacheable response, if another owner populates the shared cache while the miss is pending, the older response returns frozen `superseded` and cannot overwrite that newer entry; a `no-cache` response likewise leaves the newer entry intact while retaining its own `not-cacheable` disposition. Non-success responses, wrong MIME, empty or malformed XML, and unsafe or invalid response metadata reject with typed redacted errors. Request-ID, transport, metadata, body-reader, and parser failures retain no host cause or payload-derived message. `cancel(source)` aborts the exact active URL, `cancelAll()` aborts every cancelable preload, and a late adapter response after either successful boundary returns `canceled` without caching; cancellation before request-ID allocation omits that unavailable field from its frozen report. The final synchronous cache write is protected before invoking the cache adapter, so `cancel(source)` returns `false` once a response has crossed that boundary.

`retain(source)` returns an immutable caller lease `{ promise, commit(), release() }` over the same canonical operation. A direct `preload()` or a committed lease makes that operation durable. Only the final uncommitted lease can abort a still-pending operation before the cache-write boundary; releasing a lease never evicts a settled snapshot. `cancel(source)` and `cancelAll()` remain the explicit force controls and may still cancel otherwise durable pending work before that boundary.

React hosts may inject the same preloader through `ExpoTurboProvider.documentPreloader`. A component that calls `useExpoTurboDocumentLink(href)` is the explicit native equivalent of a semantic anchor; after that exact component renders, raw `data-turbo-preload` presence schedules the canonical cache URL through the shared preloader. The marker value is irrelevant, including `"false"`; an admitted fragment-bearing marker strips its fragment before cache reservation and transport, so fragment variants share one in-flight request and snapshot. An optional `DocumentAutomaticPreloadPolicy.canPreload(canonicalUrl)` runs after this admission: absence allows, literal `false` silently starts no cache/request work, and thrown, nonboolean, or async results fail closed through the optional redacted renderer observer. The renderer rechecks its exact live node, raw `href`, and marker admission after the policy, so policy-side mutation cannot warm the prior destination. This policy governs only render-time marker work, not direct `DocumentPreloader` calls or press-in prefetch. This render-bound design discovers initial and replacement document links, links rendered by a matching Frame response, cached-preview links, and links inserted into a rendered subtree by a Stream without guessing that every arbitrary `href` attribute is a link. Protocol `template`, Stream, and stream-source subtrees never render and therefore remain inert.

Automatic admission is intentionally bounded by the native link contract. Disabled or closest-`data-turbo="false"` links, generated-form method/Stream links, unsupported native download/non-current-target metadata, non-document Frame destinations, cross-origin/outside-root URLs, and excluded extensions schedule nothing. Exact `_top` and other targets that resolve through the current native Frame rules to the owning document may preload. Strict Mode replay and duplicate semantic links share `DocumentPreloader`'s canonical in-flight reservation. Link cleanup does not cancel shared host-owned work; runtime disposal may call `cancelAll()`. A rejected preload cannot alter rendering or visit ownership: a still-current exact link reports the typed package error, or a redacted `RequestError` for a hostile injected requester, through the optional provider error observer, while effect replay, unmount, marker removal, href change, and same-key node replacement suppress stale reporting.

`useExpoTurboDocumentLinkPrefetch(href)` separately provides the touch-native cache-warming equivalent of a browser prefetch intent. An app-owned semantic link may attach its stable fire-and-forget callback to `Pressable.onPressIn`, the callback's `.cancel()` method to `onPressOut`, and call `.commit()` as the first synchronous statement in `onPress`; the standalone Expo example uses that wiring. After full current-link admission, an optional `DocumentPrefetchPolicy.canPrefetch(canonicalUrl)` receives only the canonical URL. Absent policy allows prefetch; literal `false` is a silent no-op with no lifecycle/cache/request-ID/request-lifecycle work, while thrown, nonboolean, or async results fail closed through the optional redacted renderer observer. The policy runs before lifecycle admission and again immediately before preloader ownership, so policy- or listener-side changes fail closed. An optional `DocumentVisitController` lifecycle then receives frozen cancellable native `before-prefetch` `{ nodeKey, url }` immediately before the renderer invokes the same injected `DocumentPreloader`. Prevention is a silent no-op: it starts no preload/cache/request-ID/request-lifecycle work and cannot cancel a prior shared preload or affect a later link activation. Before work begins the renderer rechecks provider identity, live node identity, raw `href`, and full press-in admission against the captured URL, so listener mutation/removal/base/target/opt-out changes fail closed rather than warming an unobserved destination. This deliberately differs from browser Turbo: native dispatch follows full native admission, has no DOM event target/bubbling, and then rechecks; browser Turbo dispatches before root/adapter admission and captures its URL afterward. Native does not read browser document-wide `turbo-prefetch=false` metadata; a host may express global or per-location policy through `DocumentPrefetchPolicy`. Direct `DocumentPreloader` calls and automatic `data-turbo-preload` warming publish no `before-prefetch` and never consult this press-in policy; automatic warming may separately consult `DocumentAutomaticPreloadPolicy`. A successful press-in can use only the preloader's ordinary request lifecycle and never publishes document visit/history/tree ownership. An empty or case-insensitive `get` `data-turbo-method` remains prefetchable. The narrower admission rejects fragments and a same path/query, every present browser `target`/download, other method values, Stream metadata, confirmation and UJS-style metadata, cross-origin/outside-root/unvisitable locations, Frame-local destinations, and the closest `data-turbo="false"` or `data-turbo-prefetch="false"` setting. A Frame-contained link participates only when the existing native Frame target resolver promotes it to the owning document, such as explicit or inherited `_top`; that cache-only request never preloads a Frame response. A nearer non-false setting overrides an ancestor opt-out. Repeated presses share the preloader's canonical in-flight/cache result. Press-out queues release to a microtask so the normal native `onPressOut` then `onPress` sequence commits the same lease; an abandoned gesture releases only its exact uncommitted sole pending lease, without canceling direct, automatic, concurrent, or committed shared work, and cannot evict a settled shared snapshot. Preload-only injected requesters remain compatible; without `retain`, press-out remains a no-op. Current failures use the normal redacted renderer error observer, while stale or unmounted nodes suppress reporting. This is not browser hover timing, a TTL/response-reuse contract, Frame prefetch, anchor activation/scrolling, or physical-device evidence.

This API does not apply a fragment after cache warming, preload Frame responses, implement browser-hover timing/delay, or provide a TTL/request-reuse contract. Ordinary preview consumption is the separate visit-controller behavior below.

For a fragment-free ordinary top-level `advance` or history-backed `replace`, a configured `DocumentVisitController` performs one `getPreview` lookup before the canonical request. A miss, including a retained `no-preview` entry, uses the existing network-only path. A hit selects its independent clone before any same-key outgoing capture can overwrite the cache, claims the shared document destination, publishes one `started` lifecycle, captures the latest outgoing tree, commits the requested history proposal when configured, and installs the clone as a provisional whole-document tree without creating a document-autofocus claim. `DocumentSession.treeState` is a frozen `{ generation, preview }` snapshot that changes only on whole-tree replacement; `DocumentVisitSnapshot.previewVisible` projects the same flag through core and the React document binding. Tree-state notification is tracked independently from synchronous state reads, so an earlier node/disposal listener cannot absorb and hide a provenance transition from subscribers. In-place Stream, Frame, and state mutations retain the current provenance, while every ordinary `replaceTree` call installs authoritative truth and receives its own autofocus opportunity.

The preview is never a terminal cache restore. After provisional replacement, the controller immediately claims one canonical GET under the same visit/progress epoch and never captures the preview as outgoing truth. The initial cache lookup is guarded by the document-claim serial captured before lookup; the continuation records the exact preview generation and successful shared document-claim serial before tree-finalization callbacks run. A later direct tree replacement or successful document claim—including one started by cache, request-ID, node, disposal, or tree-state callbacks—therefore wins instead of being reclaimed by the older continuation. Cancellation is ignored while outgoing capture/history is protected by the destination commit, but is honored after that guard releases and before canonical ownership begins. If preview replacement commits and synchronous disposal/listener finalization reports `DocumentSnapshotPreviewCommitError`, the error remains observable while the exact still-current continuation performs its mandatory canonical GET. Redacted host-clock timer setup or cleanup failures likewise cannot suppress that request or strand the visit lifecycle.

A same-URL authoritative XML tree replaces the preview with no second history write. If the canonical response redirects to another root-visitable document, Turbo's redirected follow-up shape is preserved: the already-published requested entry remains in host history, then a fresh same-index `replace` proposal for the final URL commits before the final tree. Authoritative `4xx`/`5xx` XML likewise clears preview provenance before the lifecycle reports failure. A successful `204` or blank `201` completes without a tree, so the preview flag remains true; cancellation and typed request failure likewise leave the current provisional tree/history visible for explicit host retry or reconciliation rather than pretending cached bytes became canonical. A stale cache callback, cancellation between preview and request claim, changed tree generation, history drift, or shared-destination supersession prevents the older canonical continuation. When no request ID was allocated, that bounded cancellation returns frozen `{ source: "preview", status: "canceled", url }` evidence.

`DocumentSession.captureSnapshot(cache)` records the exact active tree under its own document URL without changing session revision or generation. `restoreSnapshot(cache, url)` returns a frozen `miss` without changing the session, or routes a fresh hit through the existing whole-tree replacement boundary. A hit therefore increments revision and generation once, disposes outgoing node identities descendant-first, gives restored nodes fresh session identities, and preserves commit-before-finalization behavior if a disposal or listener reports an error. A tree-generation change makes an older document response ineligible to commit once it settles; this manual primitive does not immediately cancel or settle an owning visit controller.

An optional `DocumentVisitController` cache captures outgoing truth for ordinary internal `advance` and history-backed explicit `replace` visits. On a preview hit, capture occurs after selecting the independent target clone and immediately before requested history plus provisional replacement; canonical revalidation does not recapture that preview. On a miss, capture occurs synchronously inside the loader's guarded final tree-commit phase, after successful redirected-response root classification and immediately before any optional history write and authoritative tree replacement, while the loader still owns the exact outgoing session. Mutations committed while fetch/body read/parse were pending are therefore included on the miss path. Valid `2xx`, `4xx`, and `5xx` XML replacements capture even when disposal/listener finalization later reports `DocumentCommitError`. Empty `201`/`204`, cancellation, transport/content-type/parse failure, initial delegation, discarded successful redirects, and refresh do not fabricate a capture on the network-only path. A root `no-cache` policy makes the attempted capture an LRU no-op; explicit restore and host traversal use the cache-first paths below.

`DocumentHistory` owns a dedicated synchronous `DocumentHistoryHostAdapter` boundary in addition to its identity/restoration ledger. Initialization happens once. Every public mutation holds one exclusive synchronous guard across validation, injected restoration-identifier generation, any host write, and internal publication; reads remain available, while an injected callback cannot reenter and publish competing history state. Managed host state validates and adopts the supplied entry without a host write or new identifier, which is a native remount-continuity rule rather than a claim that browser startup uses identical UUID behavior. Unmanaged host state is normalized to a credential-free absolute HTTP(S) URL and receives a generated restoration identifier at index `0`; the adapter must acknowledge the exact frozen `replace` before `current` is assigned and initialization returns `hostState: "replaced"`. A thrown host error or any returned value, including a Promise or thenable, is rejected and redacted before ledger publication. Failed unmanaged initialization remains retryable with a fresh identifier; a failed proposal remains exact, unbound, and retryable. The adapter must validate first, perform the exact synchronous host mutation as its final fallible operation, and return exactly `undefined`. A host that mutates and then throws or returns an invalid value is out of contract: package ledger state remains retryable, but external host state requires authoritative reconciliation rather than pretend rollback.

`adoptTraversal` is the terminal ledger-adoption step, not a cancellable or preflight callback. It binds a validated entry and derives Turbo's pop direction: a target index greater than the current index is `forward`; lower or equal is `back`. Different identifiers may share one index because Turbo replacement can issue a fresh identifier without incrementing history. One identifier may also appear at multiple URLs/indexes because Turbo's promoted Frame controller reuses its identifier across history updates. Restoration data is available only for identifiers admitted by initialization or traversal and currently admits one copied, frozen finite `{ x, y }` scroll position per identifier through `updateRestorationData`; promoted Frame entries therefore share their identifier's data, while distinct identifiers remain isolated. Traversal adoption itself performs no host write. `DocumentVisitController.restoreTraversal()` coordinates that adopted truth with the active tree, and the managed-entry subscription boundary below connects it to a host event source.

Visit proposals stage host/history coordination without changing current history. `proposeAdvance(requestedUrl)` compares the normalized requested location with the exact captured current entry: a different location proposes `push` with a fresh restoration identifier and index plus one, while the same location proposes `replace` with a fresh identifier at the current index. `proposeReplace(url)` always retains the current index. `retargetProposal(proposal, finalUrl)` preserves the original method, identifier, index, and base-entry identity while replacing only the normalized URL, so a redirect back to the referrer remains a push when the requested location differed and a same-location visit redirected elsewhere remains a replace. Issuing or abandoning a proposal does not bind its identifier or mutate current history. `commitProposal` validates the exact ledger-issued, unsettled proposal, current-base identity, and unbound identifier before host code; it then locks every public history mutator, writes the exact frozen entry and method, and performs only internal proposal deletion, identifier binding, and current-entry assignment after normal return. The host sees the old `current`. A thrown host error leaves the exact proposal/current/binding state retryable and redacts the cause. Forged, copied, retargeted-old, double-committed, concurrently stale, reused-identifier, invalid, and safe-integer-overflow cases make no host call.

When configured with `history`, `DocumentVisitController` uses this proposal path for top-level `advance` and explicit `replace` visits. Before request ownership changes it canonicalizes the active document URL through the same credential-free HTTP(S) resolver, requires `history.current.url` to match it, creates the action-specific proposal, and rechecks both identities after restoration-ID generation. Default-port and other URL-parser-equivalent spellings therefore align without weakening origin or credential checks. `advance` selects push versus same-location replace; an explicit `replace` is rejected without history and always proposes a fresh restoration identifier at the current index. On a preview miss, an admitted redirect retargets that uncommitted proposal without changing its method, identifier, or index; the guarded final path captures outgoing truth, commits the acknowledged history write, and replaces the tree. On a preview hit, the requested proposal commits before the provisional tree so interactions from that tree remain history-aligned. A same-URL canonical response writes no second entry; a redirected canonical tree prepares and commits a fresh same-index `replace` against the preview entry, matching Turbo's follow-up visit rather than rewriting already-published history. Empty, failed, or canceled revalidation cannot roll back the requested host write and leaves explicit preview state. Current-document refresh remains a separate snapshot-free `replace` proposal for canonical active truth and never consumes a preview. A fragment-free root-visitable explicit `replace` whose requested and active URLs share a pathname is the bounded exception: it skips preview, reads exact morph/scroll metadata from the active root, uses the canonical GET, and can apply compatible-root morph plus post-render root reset.

An exact explicit `restore` passed to `DocumentVisitController.visit` uses `proposeAdvance` but requires the requested location to remain root-visitable, fragment-free, and aligned with the configured current history entry. A different requested URL therefore proposes `push`; the exact current URL proposes same-index `replace`; both receive a fresh restoration identifier. The host may supply a validated direction (`back`, `forward`, or `none`, default `back`) and an optional finite `{ x, y }` root scroll snapshot in `restorationData`. Unlike `advance`/`replace`, an initially unvisitable explicit restore fails closed rather than delegating to a host router; a cache-miss successful redirect likewise fails closed when response-root policy makes its final location unvisitable. With a configured cache, the target lookup is unconditional, so retained `no-preview` snapshots remain eligible. A hit clones the target before claiming ownership, acknowledges the ordinary visit lifecycle, attempts the latest outgoing snapshot under normal cache policy (`no-cache` skips storage), commits the exact proposal, then replaces the document—with no request-ID allocation or GET. Its frozen result is `{ source: "snapshot", status: "restored" | "canceled", url }`. A miss performs one guarded document GET through the parsed-response path above; an admitted successful redirect retargets the original proposal to the final URL without changing its method, identifier, or index. The admitted root scroll is offered only for a successful document and only after exact React acknowledgement, through the same `DocumentHistoryScrollAdapter` ordering used by host traversal. Authoritative redirected or unredirected `4xx`/`5xx` XML is outside that successful-redirect root-reproposal branch and remains commit-capable before the lifecycle reports failure, without applying the supplied scroll. History and visit-attempt drift are rechecked before ownership and again around snapshot/history commitment. Cancellation, supersession, empty responses, transport/MIME/parse or root-policy failure, snapshot failure, and host-write failure publish neither the proposed entry nor a replacement tree. A finalization failure after history and cached-tree replacement reports committed truth. This explicit action is distinct from `restoreTraversal(entry)`: the former creates and writes a fresh entry using host-supplied presentation metadata, while the latter consumes an entry and ledger-owned restoration data after host history already moved.

`restoreTraversal(entry)` is the host callback boundary after native history has already moved. It first requires either an aligned active document/history pair or the exact still-pending prior traversal entry, rejects cross-origin and fragment-bearing locations before ledger adoption, then delegates canonical validation and `back`/`forward` derivation to `adoptTraversal`. The adopted canonical entry is never written back to the host. With a configured cache, lookup uses unconditional restoration semantics rather than `getPreview`, so a retained `no-preview` snapshot is eligible. A hit claims the shared document destination, acknowledges `started`, captures the latest outgoing tree under the final commit guard, rechecks host history after that fallible boundary, retargets the fresh cached clone to the exact traversed URL, replaces the tree, and completes without a request. Lifecycle cancellation may stop the claimed cached restore before the final guard. A miss performs one ordinary GET without a history proposal; its final parsed candidate must be unredirected, match the adopted URL, and still be the exact current history entry before and after outgoing capture and tree replacement. A newer traversal supersedes older cached or network work. Empty `201`/`204`, transport/MIME/parse failure, redirect, snapshot failure, and history drift leave the host/ledger entry adopted because host traversal is authoritative; they do not fabricate rollback, and a later host entry can recover from drift. The frozen result reports the canonical entry, direction, restoration data, source (`snapshot` or `network`), and the source-specific terminal report.

For fragment-free host traversal and explicit restore, an admitted root position is eligible when the controller has a live `DocumentVisitLifecycle` and React acknowledges the exact committed document. A cached hit, or a successful admitted network document, passes that position once to `ExpoTurboProvider.documentHistoryScroll`, whose synchronous `DocumentHistoryScrollAdapter` must return a boolean from `canRestore()` and `undefined` from `restore(position)`. Restoration runs after document autofocus and before `load`. No position, an unavailable adapter, headless/stale/canceled/superseded/unacknowledged rendering, and empty/error outcomes do nothing; adapter failures are redacted render errors and never roll back the committed tree. The position is root-scoped: promoted Frame entries that share one identifier share one position, while Frame-local scroll has no contract. Ordinary advance/replace visits, anchors, nested containers, and capture/persistence remain outside this contract.

`subscribeDocumentHistoryTraversal(source, controller, { onError, onResult })` is the host-neutral managed-entry event boundary. Subscription setup, invalid cleanup contracts, and cleanup failure throw synchronous redacted state errors. The adapter does not merge the history write boundary with traversal delivery or prescribe one router representation.

## Document visit lifecycle and native accessibility

The optional `DocumentVisitLifecycle` exposes a frozen notification-only `before-cache` event immediately before an ordinary cacheable outgoing document is captured by a controller-owned network visit, cached preview, explicit cached restore, cached or network host traversal, successful safe-GET document form response, or promoted Frame history plan. Dispatch runs outside the destination commit guard so synchronous listener mutations are included in the captured tree; exact ownership, history, and tree-generation checks then run again before guarded snapshot/history commitment and replacement-tree application. Direct promoted Frame GETs dispatch before publishing the requested Frame `src` or beginning fetch. Successful promoted Frame forms dispatch after Expo XML response admission but before publishing the final response `src` or buffering the body; an unsafe promotion clears older cache entries after the event and retains only the newly captured outgoing history snapshot. The event carries no detail, uses stable listener snapshots, requires an exact synchronous `undefined` return, and isolates observer failures through the lifecycle's redacted error surface. A listener that changes the outgoing document to `no-cache` still receives the event, after which snapshot storage rechecks policy and skips the cache write. Refresh, preload/manual capture, initially `no-cache` documents, failed/empty/Stream form responses, unsafe history-neutral document/forms, and history-neutral Frame work emit no event. This is a synchronous native equivalent and does not claim browser Turbo's next-task clone delay or renderer-dependent promoted-Frame visual timing.

`reload` is a frozen notification-only native recovery signal, not a browser reload. An active `DocumentVisitController` GET emits `{ cause: "transport" | "content-type", reason: "request-failed" }` only after it publishes `failed` when its direct fetch/body transport rejects or MIME admission fails; it carries no raw error, URL, body, or response headers. The controller rechecks its exact failed epoch before dispatch, so a listener may begin a newer visit without stale state mutation. Current-document refresh and canonical preview revalidation are ordinary active GETs and remain eligible. Request-lifecycle default prevention, explicit cancellation, supersession, explicit restore, host traversal, parse/policy/commit failures, committed XML `4xx`/`5xx`, form/Frame work, and preload emit no reload. The event neither retries nor invalidates/reloads a native screen, and it makes no DOM, page, or physical-paint claim.

`DocumentRequestLoader` keeps `beforeCommit` as the fallible parsed-candidate admission phase. The optional `beforeTreeCommit` phase is narrower: it runs only for an admitted parsed tree after the loader's last ordinary ownership check, receives the exact same frozen candidate, and executes synchronously under the session-shared destination-ownership guard. While that callback is active, another document or Frame request cannot claim the session and exact-owner cancellation returns false. Returning any value, including a Promise/thenable from an async callback, is a protocol error; throwing leaves the active tree unchanged and releases ownership for retry. The callback must itself be atomic-on-error and must not mutate the document session or reenter package controllers. The guard clears immediately before `replaceTree`, so disposal/listener finalization observes the new tree and may start newer work without the outer loader later detaching it. Empty, discarded, canceled, stale, transport, MIME, and parse outcomes never enter this phase. The optional parsed-visit history coordination above uses this exact linearization hook.

`DocumentVisitController` gives one mounted document session an identity-safe observable GET lifecycle. Its immutable snapshot moves from `initialized` to `started`, then to exactly one of `completed`, `failed`, or `canceled`. `previewVisible` independently reports whether the active whole-document tree remains provisional and may therefore remain true after any terminal status; it clears only when authoritative whole-document truth replaces that tree. The controller publishes `started` only after a document lease is installed, including the preview lease when one exists, so a blocked or synchronously superseded claim never creates false busy state or displaces an incumbent lifecycle. Previous-timer cleanup and scheduling run under exact epoch checkpoints, with the old handle detached before host clock code, so reentrant clock cleanup cannot overwrite a newer visit. Timer setup/cleanup failures report redacted state errors while transport and terminal lifecycle settlement continue. `busy` becomes true at the acknowledged start. `progressVisible` becomes true only after the injected clock reaches `DOCUMENT_VISIT_PROGRESS_DELAY_MS` (500 ms) and spans preview plus canonical revalidation without a second start. Only a successful exact loader cancellation or the exact post-commit pending preview continuation changes lifecycle immediately; stale/not-owned and commit-protected attempts are no-ops. After a real abort the controller rechecks the visit epoch before publishing, preventing synchronous abort listeners from starting newer work that the outer cancel then clears.

A committed success document, `204`, or empty `201` completes the visit. A valid `4xx` or `5xx` XML error document commits first and then fails the visit. Network, MIME, parse, and pre-commit policy errors fail while preserving the prior tree. Explicit cancellation and supersession are `canceled`, never request errors. Epoch, request-owner, session-generation, and document-claim-serial checks prevent stale requests or cleared timers from publishing into a newer visit. A session finalization error after a successful authoritative commit remains observable while the lifecycle records `completed`; the equivalent error after an authoritative `4xx`/`5xx` commit records `failed`. Preview finalization error observability does not terminate a still-current canonical revalidation.

The sole XML root may declare `data-turbo-root`; absence means `/`, and relative values resolve against that document's URL. Top-level visitability follows Turbo 8.0.23's exact origin-plus-path prefix algorithm and fixed case-sensitive unvisitable-extension set. A cross-origin proposal delegates to `NavigationAdapter.openExternal`; a same-origin proposal outside the root or carrying an excluded extension delegates to `NavigationAdapter.visit` with the admitted action, `advance` by default. Runtime action admission accepts only exact `advance`, `replace`, and `restore`; a forged value rejects before URL classification or request ownership. A visitable explicit `replace` or `restore` additionally requires aligned initialized history. Unvisitable `advance`/`replace` may delegate before document ownership changes, while unvisitable `restore` fails closed because the package cannot map an arbitrary restoration URL to native host history safely.

After a successful same-origin redirect, the controller reclassifies the final URL against the parsed response document's root before tree replacement. An unvisitable candidate is discarded and the final URL is delegated with `replace`; a visitable candidate commits. A nonredirect response may redefine its root and commit normally, while redirected `4xx`/`5xx` XML remains authoritative error output and is not reproposed. This is a native-equivalent ordering: unlike browser Turbo, the discarded response is not rendered before host delegation, so adapter failure leaves the prior tree active. Cross-origin final redirects remain a typed transport rejection because the loader refuses to admit a final URL on another origin.

React hosts inject the controller through `documentController`, may supply a `documentComponent` boundary, and can read the nearest binding with `useExpoTurboDocument()`. The binding exposes the frozen controller snapshot, including `previewVisible`, plus a frozen `{ busy }` accessibility state for an actual native boundary. The controller subscribes to whole-tree provenance only while it has state observers, and its state getter synchronizes the same frozen session flag for headless reads; an authoritative replacement by another package controller therefore cannot leave a subscribed boundary claiming a stale preview. Controller ticks preserve the boundary and do not remount unchanged registered children. The controller remains host-owned across React unmount and Strict Mode effect replay; React subscribes and reports errors but never silently cancels it. State and error observers are snapshotted and isolated; hosts can provide `onObserverError` to surface observer failures without changing the visit's request result, and an unhandled observer failure is rethrown on a microtask.

An optional `documentAnnouncements` adapter receives only a frozen non-`initialized` `{ status }` event after the corresponding document-boundary React commit. React baselines the mounted controller and delivers one later status transition globally per controller: a progress tick that retains `started` does not repeat delivery, while a completed, failed, canceled, or later terminal-to-`started` state may deliver. Initial or pre-mounted state, stale snapshots, and Strict Mode/multiple-provider replays do not announce. The package supplies no user-facing copy or platform accessibility import and never exposes URLs, response bodies, or server text; the host localizes and delivers the status. Synchronous or asynchronous delivery failures reach the provider error callback without changing the visit result or blocking later transitions. Physical screen-reader evidence remains pending.

The controller accepts initial `advance` visits, history-backed explicit `replace` and `restore` visits, bounded current-document Refresh Stream morphs/replacements, and same-origin fragment-free host traversal entries through `restoreTraversal`. Fragment-free ordinary `advance`/`replace` may render a previewable cached tree before canonical GET; explicit restore and traversal instead use unconditional cache hits as final restoration truth. Initial visits may delegate an unvisitable successful redirect through host navigation with `replace`; explicit-restore successful fallback redirects and traversal fallback redirects instead fail closed when they cannot preserve their distinct history contracts. Authoritative redirected `4xx`/`5xx` Expo XML remains commit-capable because it is not successful-redirect root reproposal. A later host entry can recover after concurrent traversal history drift, and lifecycle cancellation can stop a cached restoration before replacement. The host still owns history initialization plus concrete router state/event mapping; `subscribeDocumentHistoryTraversal` only consumes the resulting managed entries. The standalone Expo example supplies one such mapping without changing this host-neutral API. Exact fragment-free same-path `replace` morph/root-scroll metadata is supported; fragment/anchor restoration, Frame previews, root scroll capture/persistence, Frame/nested scroll-state application beyond that bounded root reset, the remaining lifecycle-event families recorded above, browser progress-bar animation, and physical-device accessibility evidence remain unsupported.

## Injected Cable stream sources

`<turbo-cable-stream-source>` is a zero-visual protocol node. A host creates one `CableStreamSourceRegistry` for a `DocumentSession`, supplies its own synchronous `CableAdapter` plus required error observer, and injects the registry through `ExpoTurboProvider.streamSources`. The renderer retains only exact source nodes that are part of the rendered application tree. Merely appearing in the tree's source index does not activate a subscription, and sources inside `template`, `<turbo-stream>`, or another source subtree remain inert. The provider releases its rendered leases but does not own the injected registry; the host calls `registry.dispose()` when the `DocumentSession` lifetime ends.

Every source requires a nonblank `channel`. `Turbo::StreamsChannel` additionally requires a nonblank `signed-stream-name`; custom channels may omit it. Extra `data-*` attributes become deterministic snake-case Action Cable identifier parameters. Reserved/colliding/malformed parameters fail closed, dataset fields are sorted, and sharing within one registry is keyed by the complete canonical JSON identifier rather than channel alone. Rendered duplicate sources therefore share one transport subscription while retaining independent exact-node leases. Descriptor changes rebind synchronously; an invalid descriptor disconnects only that source, and a later valid correction can reconnect it. The final lease stops accepting messages immediately, while transport cleanup is deferred by one microtask so Strict Mode replay or another provider can reacquire without transiently unsubscribing. Logical removal/replacement invalidates the exact source synchronously.

The Rails gem's optional protected descriptor uses its own Channel, a gem-verifier token in `signed-stream-name`, and a client-visible `data-grant`. The native registry treats those values as opaque identifier data. On the server, that Channel verifies the token with the separate verifier, authorizes the decoded canonical Expo stream with the host subject and grant, then subscribes to the opaque token topic rather than the public Expo stream name. The generic `Turbo::StreamsChannel` cannot verify that token. Authorization is admission-only, so the host must terminate the affected subscription or connection and require a fresh admission when a grant expires or is revoked. Credential transport, grant issuance/expiry, host resource policy, and identity-triggered client reconnection remain host responsibilities.

`connectionSnapshot` exposes a frozen, revision-cached, node-key-sorted list of redacted `{ nodeKey, state }` records. `subscribeConnection(listener)` runs only after a semantic snapshot change, so listeners reread the getter and receive no immediate callback. A source using a new transport begins `connecting`; one that joins a shared transport adopts its current state, and a synchronous callback may settle either before the first observation. Active adapter callbacks may report `connected`, `reconnecting` (`disconnected(true)`), `disconnected`, or `rejected`. Shared canonical transports update every exact source they serve. A valid descriptor rebind starts a new transport; an invalid descriptor is `disconnected`; removal/disposal removes the record; and stale transport callbacks cannot update a later source/transport binding. Connection observer exceptions are isolated and reported as redacted subscription errors. Snapshots contain no channel, canonical identifier, signed name, grant, message, or raw transport failure.

String messages from every active transport enter one registry-owned synchronous queue and use the same `dispatchTurboStreamFragment` path as HTTP and embedded Streams. A malformed message or observer failure is isolated without poisoning later delivery, and a removed transport's queued or late callbacks cannot mutate the document. Subscribe, unsubscribe, invalid-return, non-string-message, and dispatch failures are exposed only as redacted `SubscriptionError`s; canonical Action Cable identifiers, signed names, grants, message bodies, and raw adapter failures are never attached.

The adapter surface exports a strict `actioncable-v1-json` codec for opaque subscribe/unsubscribe commands plus welcome, ping, disconnect, confirmation, rejection, and string-delivery server frames. It freezes admitted frames and rejects malformed envelopes without retaining raw payloads. `resolveActionCableEndpoint` is the pure admission primitive: it converts one credential-free HTTP(S) origin plus an explicit same-origin mount path into a `ws:`/`wss:` URL, rejecting userinfo, query/fragment, origin paths, and authority-changing or normalizing mount paths. It does not choose a host, carry tickets, or create a socket. `ActionCableV1WebSocketAdapter` is the bounded socket primitive: a host supplies that absolute `ws:`/`wss:` URL, an injected native-WebSocket wrapper, and a redacted error observer. It creates at most one socket at a time, requires the exact negotiated subprotocol, waits for `welcome` before subscribing, reports `connected(false)` only after an exact confirmation, shares one server subscription per opaque identifier, retries every active unconfirmed group through one injected 500 ms timer until confirmation, and routes deliveries only to active exact identifiers. It clears that timer after the final confirmation, rejection, unsubscribe, socket detachment, or disposal, and closes the socket when its final local subscription leaves. An optional heartbeat monitor records every valid server frame through the supplied scheduler and time source; after more than six silent seconds it performs one replacement through the same `disconnected(true)` and re-confirmation path. A replacement that remains silent does not loop; the next valid frame rearms the bounded monitor. A valid server `disconnect` with `reconnect: true` uses that same replacement path: it detaches the old generation before callbacks, opens one replacement socket, resends every still-active exact identifier only after its `welcome`, and emits `connected(true)` only for records that had already confirmed before the directive. Records awaiting their first confirmation still receive `connected(false)`. Late old-socket events are inert. A close/error, malformed frame, unsupported protocol, `disconnect` without reconnect, or command failure disconnects active callbacks with `false`, reports a redacted `SubscriptionError`, and makes that adapter terminal. Replacement construction/listener failure settles its active callbacks with `false` and reports the redacted failure; it never uses generic connection-retry backoff or automatic socket retry.

The registry deliberately does not synthesize `connected` attributes or own the socket implementation. Its connection snapshot reports adapter callbacks; it is not a transport health guarantee, reconnect policy, or authentication mechanism. A host may inject `reconnectRefresh`, a `CableReconnectRequester`, to reconcile an active scope after recovery: after the first still-active `connected(true)` callback in one bounded replacement cycle, the registry passes one frozen `{ baseUrl, scroll: "preserve", sourceKeys }` request. `sourceKeys` contain only active logical source-node keys from that cycle, never an Action Cable identifier, signed stream name, grant, message, or raw error. Initial connections, terminal transitions, a replacement that never re-confirms, stale/rebound/disposed transports without a successful replacement, and ordinary Turbo Stream `refresh` never take that path; multiple active identifiers in the same cycle still produce one handoff. With `DocumentRefreshController`, the request is coalesced and runs only if the captured URL still owns an idle current document. A host may wrap it in `DocumentReconnectReconciler` to retain one recovery request that arrives while an owning document visit is active; when that visit sequence next becomes idle—completed, canceled, or failed—the wrapper hands the latest frozen request back to the refresh controller, which repeats its existing URL-ownership check. A changed document or disposal produces no extra GET; a superseding active visit keeps the same one request deferred until it settles.

`FrameReconnectReconciler` is the separate source-aware opt-in. It snapshots each listed source node's exact identity, waits while any still-active captured source is `reconnecting`, and waits through an active document visit or busy eligible Frame. A connected document-level source wins and delegates one `{ baseUrl, scroll: "preserve" }` refresh; otherwise it finds only already-mounted exact Frame controllers, selects active outermost eligible Frames in document order, and reloads them serially through their own request path. It revalidates source identity, connection state, Frame identity, mounted-controller identity, connection, disabled state, and `src` before every reload. Removed/replaced/rejected/disconnected sources and missing/unmounted/disabled/source-less Frames silently drop without promoting a root refresh; a later valid nested Frame remains eligible if its outer Frame is not. Reload failures are redacted and isolated, while a lifecycle-prevented transport failure remains handled by its existing request lifecycle. This recovery does not create controllers, retry sockets, replay messages, authenticate subscriptions, or alter ordinary Stream `refresh` semantics. It has focused public core coverage and is wired into the standalone Rails Cable panel, whose sole fixture source lives inside an eager Frame; the focused panel runtime test covers the Frame-only recovery path. The bounded socket adapter owns no credentials/tickets, AppState/network behavior, generic connection retry/backoff policy, protected resource authorization, Rails/Redis integration, or physical-device proof; those remain host/transport work and are not claimed by this slice.

## Document refresh Streams

The built-in `refresh` Stream action ignores `target`, `targets`, and template content. It captures the active document URL plus optional `method`, `request-id`, and `scroll` attributes, then delegates to the host-wired `DocumentRefreshController`; dispatch itself never performs a nested fetch while a Stream response still owns its destination lease. A missing controller or active document URL is an isolated action error and does not prevent later sibling Stream actions from running.

Turbo 8.0.23 selects refresh morphing only for exact lowercase `method="morph"`; an absent, blank, differently cased, or arbitrary method value uses ordinary replacement. The native controller accepts that exact method only for one bounded success-only path: after the 150 ms trailing debounce, request-ID suppression, active-URL/idle-visit checks, and ordinary GET admission, it preflights before snapshot/history commit callbacks. The response must contain exactly one compatible nonpermanent ordinary application root with the same QName plus one valid reconciliation plan. That retained root reconciles an added, changed, or removed `id`; a new root ID already owned by an active descendant fails preflight. Exact-shape application identities and stable-ID permanent application descendants may move among ordinary parents inside that root; incompatible ordinary wrappers may remount around retained stable descendants, while same-parent anonymous wrappers use persistent descendant-ID intersection before the state-free exact-QName ordinal fallback. Current permanent descendants remain opaque even when the incoming marker is absent, directly omitted permanent children remain in place, and incoming-only markers are admitted normally. An outermost mounted `turbo-frame[src][refresh="morph"]` with an absent or source-compatible incoming peer retains its exact wrapper and children, then reloads after the committed render; nested eligible Frames remain owned by that outer cascade. Changed-source or plain Frames and Cable sources reconcile structurally. Permanent roots/ancestors, protocol reparenting, malformed or incompatible permanent identities, duplicate/out-of-subtree IDs, omitted unmatched permanent-bearing ancestors, and every invalid plan fail before history, tree, revision, or disposal changes without a replacement fallback. A successful morph uses the existing snapshot-free refresh history `replace` proposal, creates a fresh authoritative generation, emits the logical page-level `morph`, reports `renderMethod: "morph"`, and deliberately suppresses document autofocus. Authoritative `4xx`/`5xx` XML always uses ordinary replacement and reports `"replace"`; every other method value uses ordinary replacement.

For a successful current-document refresh, exact lowercase `scroll="preserve"` consumes its scroll intent without a host call. An absent, blank, differently cased, or other value instead requests one top reset from the optional synchronous `DocumentRefreshScrollAdapter` (`canReset()` then `reset()`) only when the `DocumentVisitController` has a `DocumentVisitLifecycle` and React has acknowledged the exact committed document generation. No adapter, an unavailable adapter, headless rendering, a failed/superseded/stale acknowledgement, an empty response, or authoritative `4xx`/`5xx` XML consumes a no-op; adapter faults surface as redacted render errors without rolling back the committed tree. This Refresh-specific path is separate from bounded traversal restoration; it does not resolve anchors, discover nested containers, capture or persist native scroll, interpret page metadata, or prove physical scroll completion.

A separate bounded page-refresh path applies only after normal root-visitable admission when `DocumentVisitController.visit()` receives explicit `action: "replace"`, the active and requested canonical URLs have the same pathname, and the requested URL has no fragment. Before the canonical GET, the controller reads only the active tree's sole XML root: exact lowercase `data-turbo-refresh-method="morph"` selects the same success-only compatible-root morph admission described above, and exact lowercase `data-turbo-refresh-scroll="preserve"` suppresses the root reset. Absent, invalid, differently cased, whitespace-padded, or nested values default independently to `method: "replace"` and `scroll: "reset"`. This path bypasses a snapshot preview, retains normal replace-history ownership, and still requires the canonical GET. A successful compatible morph reports `renderMethod: "morph"`; authoritative `4xx`/`5xx` XML always replaces, and reset applies only after React acknowledges the exact committed generation. It does not cover fragments, different-path visits, `advance` or `restore`, Frames, nested-scroll discovery, generic route/direct-link behavior, or browser morph/scroll semantics.

The standalone gallery exposes one intentionally narrow fixture at `/demo/linked?refresh=scroll`: it begins with enough document content to put the gallery Refresh control below the fold, then its explicit default Refresh Stream returns distinct canonical XML. The example registers only its owning root `ScrollView` with `DocumentRefreshScrollAdapter`; focused runtime coverage proves the acknowledged replacement invokes that registered root reset, and the iOS Simulator/Expo Go run visibly returns to the top with the committed document. This does not establish generic/nested/anchor scroll behavior, physical-device completion, or Android interaction.

When `DocumentVisitController` has aligned `DocumentHistory`, `refreshCurrent` prepares a `replace` proposal for the exact canonical current URL before request ownership while intentionally omitting outgoing snapshot capture, matching Turbo's `shouldCacheSnapshot: false` refresh visit. A committed successful bounded morph or ordinary `2xx`/`4xx`/`5xx` replacement writes a fresh restoration identifier at the current index before its tree mutation; an admitted same-origin redirect retargets that same proposal to the final URL. Turbo performs that redirect through a second replace visit, while this native history boundary intentionally normalizes the observable result into one atomic final-URL replace write with one fresh identifier. Empty responses, cancellation, supersession, transport/MIME/parse failure, discarded redirects, concurrent history movement, and host-write failure leave the prior history entry and tree authoritative. Without configured history, the same refresh request remains an ordinary history-neutral document replacement.

Each document session retains the newest 20 request IDs in insertion order. Document GETs, Frame and recurse GETs, and canonical form submissions record their generated ID immediately before transport; a caller-scoped `FormRequestExecutor` can be given the same tracker explicitly. A refresh carrying one of those IDs is suppressed when the debounce fires, preventing the originating request from refreshing itself twice. This is loop suppression, not request replay, cache restoration, broader morph preservation, or missed-Cable-message recovery.

## Document-tree link activation

`useExpoTurboDocumentLink(href)` lets an app-owned registered component expose a plain native link without adding React Native or a host router to the package. The hook requires the host-injected `DocumentVisitController` and returns a stable activation callback without subscribing the component to visit-state ticks. An ordinary same-origin top-level activation delegates the unchanged `href` to that shared controller, so request headers, cancellation, response classification, and latest-visit ownership use the document GET contracts above. A link-only resolver separately admits normalized `mailto:` and `tel:` URLs; the HTTP(S)-only resolver used by document, Frame, and form fetch paths is unchanged.

As a narrow native equivalent, the hook also admits a same-document anchor request only when an enabled exact-live top-level ordinary application link has a nonempty once-decoded fragment and its canonical origin, path, and query exactly equal the active document. The decoded value must be a nonblank, control-free exact ID of a current top-level ordinary application element. Frame ancestry, any `data-turbo-frame` value (including blank or `_top`), `data-turbo-action`, generated-form/Stream/confirmation/UJS metadata, closest `data-turbo="false"`, unsupported browser target/download metadata, missing/stale source or target, and a busy document all fail closed. The ordinary cancellable `click` event receives the canonical fragment URL; prevention returns the existing frozen link-canceled result. After a noncanceled listener, the hook rechecks the exact raw source, document URL, admission metadata, and target before it invokes `ExpoTurboProvider.documentAnchorScroll`. That `DocumentAnchorScrollAdapter` invokes `scrollTo(id, "start")` synchronously and must return exactly `undefined`; throws, values, Promises, and thenables reject with a redacted state error after consuming any rejection. A successful request returns frozen `{ kind: "anchor", status: "requested", targetId, url }` evidence. It starts no fetch, document/Frame visit, history/router/tree mutation, `before-visit`, `visit`, `render`, or `load` lifecycle. The standalone Expo gallery wires only its direct `#native-anchor-target`: its example-owned adapter adds the rendered document root, document-content, and target layout offsets before it scrolls the owning root `ScrollView`. Focused adapter/runtime coverage proves the request begins neither a fetch nor a Router-history write, and an iOS Simulator/Expo Go flow visibly reaches the target. It does not claim physical completion, URL/history mutation, cross-document or route fragments, Frame/nested anchors, target discovery, or alternate alignment.

This package contract still does not supply generic host direct-link fragments. Separately, the standalone example has one native host bridge for an initial or later Expo Go `exp:`/`exps:` URL. On cold launch it validates the raw `/--/` path and ordered query against the resolved unmanaged catch-all route, decodes one nonblank control-free hash ID, and lets the existing guarded bootstrap commit that route's document before deferring the root-only anchor request until layout. Its cold `/demo/linked#linked-native-anchor-target` proof therefore performs one canonical document GET and same-key history repair before it scrolls the linked target. A later event remains same-document only: if Expo Router has stripped the current opaque metadata, the example restores that exact entry in place before scrolling and starts no document fetch or history transition. iOS Simulator/Expo Go evidence covers `#native-anchor-target` cold and from a running gallery plus the cold linked target; Frame/nested targets, later cross-document events, Android interaction, physical-device, and release-build evidence remain outside that proof.

Unless the closest `data-turbo` setting opts out, every HTTP(S) interactive link is classified by the active document root before Frame capture. Cross-origin, outside-root, and excluded-extension links delegate directly through the host navigation adapter even when authored inside a Frame, matching the browser link observer's rejection before link/visit lifecycle. A visitable activation emits the semantic click above, then an activation inside a Frame passes the unchanged `href`, nearest Frame ID, and optional `data-turbo-frame` value to the host-injected `FrameControllerCollection.visit` contract. That is the same registry-owned path as a programmatic Frame visit, so Frame-default/element target precedence, current/named Frames, `_self`, `_parent`, `_top`, URL admission, controller reuse, request cancellation, and response ownership cannot drift into a second React implementation. A missing named target inside a Frame falls back to the current Frame; a disabled current/named Frame or unavailable `_parent` promotes through the shared document coordinator. Outside a Frame, a non-`_top` target identifying an active enabled Frame enters the shared Frame visit path and applies its target rules. Top-level `_top`, missing, and disabled targets follow ordinary top-level activation. Frame and document controllers keep independent request ownership.

Hosts inject `navigation` into `ExpoTurboProvider`; ordinary root/cross-origin activation passes that adapter into the host-owned document controller, while explicit opt-out uses it directly. A credential-free cross-origin HTTP(S) URL awaits `openExternal` with the normalized absolute URL. Normalized `mailto:` and `tel:` URLs also await `openExternal`, return a frozen scheme-identifying result, and delegate before document/Frame target capture, so top-level, Frame-contained, named-Frame-targeted, and opted-out links share one external path. The closest `data-turbo` setting wins for HTTP(S): exact `false` delegates a same-origin URL through `visit(url, "advance")`, delegates a cross-origin URL through `openExternal`, and a nested non-`false` setting restores normal Frame or document capture inside an opted-out ancestor. The Frame registry receives the same `NavigationAdapter` for external HTTP(S) visits and the shared document controller for same-origin `_top` or promoted visits. Successful delegation returns a frozen `status: "delegated"` result identifying external, scheme, or owning-host navigation. A missing or failing adapter rejects the activation Promise and never starts, supersedes, or cancels a document request.

The activation callback is bound to the exact registered protocol node and raw `href` that created it. It rechecks node identity and the raw `href` at press time, then fails closed after replacement/removal or a Stream-mutated URL even if a deterministic key is reused. On that exact live node, the presence of `disabled` returns a frozen `{ kind: "disabled", status: "ignored" }` result before URL resolution, root classification, Frame capture, opt-out delegation, fetch, or navigation, leaving any current request owner unchanged. Registry hosts can decode XML presence attributes with `presenceCodec`; every authored value, including the empty string and `"false"`, decodes to `true`, while absence is left to the component schema default. Turbo 8.0.23 treats `data-turbo-confirm` alone as inert during plain-link activation, so that metadata follows the same document, Frame, delegated, disabled, and request-supersession path as an otherwise identical link. Its click predicate admits `target` only when absent, empty, or exactly `_self`; those values retain the current native document, Frame, scheme, or opted-out path. Every other browser target and any present `download` attribute instead uses the optional `ExpoTurboProvider.documentLinks` adapter because React Native has no implicit browsing-context or download fallback. The package resolves the authored URL against the active document, sends a frozen `{ target, url }` or `{ filename?, url }` request, and returns frozen delegated evidence without click/visit/request ownership; `download` wins when both attributes exist. An empty download value omits the suggested filename, while a nonempty value is preserved exactly for host policy. Missing adapters fail closed and adapter faults become redacted state errors. Link-owned action recognition is exact: only lowercase `advance`, `replace`, and `restore` are actions. A present blank, invalid, differently cased, or whitespace-padded value is a non-action that retains the ordinary document, Frame, scheme, root-delegated, or opted-out path and, like Turbo's first-value lookup, masks Frame-level action inheritance. When link metadata is absent and the resolved destination is a Frame, that exact destination Frame contributes an action only for lowercase `advance`, `replace`, or `restore`; invalid Frame values remain ordinary loads. `_top` and external paths do not inherit. Turbo does not consume action metadata on scheme, root-unvisitable, cross-origin, opted-out, browsing-context, or download links, so their native browser-equivalent delegation likewise ignores it. On a visitable top-level document path, exact `advance` enters the existing advance visit and exact `replace`/`restore` use the same owning-document path when the shared controller has aligned history; a historyless document `replace`/`restore` rejects before request ownership. A recognized explicit or inherited action captured by a Frame passes through the shared Frame registry: exact local `advance` and `replace` use the destination controller's configured `FrameHistoryCoordinator`, while `_top` promotion remains document-owned and local `restore` remains unsupported. Fragment-bearing URLs outside the bounded same-document anchor path, credential-bearing, malformed, and unsupported-scheme URLs plus the nonstandard `confirm`, `method`, or `stream` aliases reject instead of being silently reinterpreted. Upstream confirm metadata also suppresses hover prefetch; native press-in prefetch follows that suppression, while browser-hover behavior remains unsupported. The example registry supplies the actual `Pressable`, link accessibility role, authored-plus-pending disabled state, styling, and visible failure handling; those remain host responsibilities.

When the optional `documentPreloader` is present, the same hook also owns automatic marker discovery after render. Unlike a DOM query, this requires no protocol tag name or registry coupling: calling the hook declares the component's semantic-link role. Marker presence is re-evaluated with the live raw node and canonical href, so adding/removing the marker, changing the link URL, replacing the node, replacing the whole document, or rendering a newly inserted link can start or suppress work. Only marked semantic links subscribe to the session-wide revision signal: this keeps subscription count linear in marked links while re-evaluating ancestor opt-out and document-versus-Frame ownership after any structural mutation, including a newly inserted target Frame outside the link's ancestor chain. A rendered link must remain an enabled, navigable, root-visitable document destination under the native target rules; a fragment is normalized away only for the cache warming request, not for the separate bounded same-document anchor request, preview, or history behavior. This intentionally refuses native metadata whose activation is unsupported even though browser Turbo's marker scanner may fetch some target/download anchors that its click path would not follow. Link actions and `data-turbo-confirm` do not suppress an otherwise eligible marker; method/Stream presence does.

When either `data-turbo-method` or `data-turbo-stream` is present on an otherwise root-visitable and navigatable link, the optional provider `formLinks` controller owns Turbo's temporary-form path before ordinary Frame/document GET capture. The controller preserves the exact live link as its proposal/activity owner, removes the URL query from the generated action, and converts decoded query pairs to ordered hidden string entries. Repeated and empty values remain, empty names are omitted, and ASCII-case-insensitive `_charset_` names become `UTF-8`. A nonempty method is copied; blank or invalid values therefore plan GET, exact lowercase `dialog` fails closed, and POST/PUT/PATCH/DELETE use direct HTTP verbs with `_method` retained as ordinary ordered data. Stream presence negotiates Stream responses even when its authored value is `"false"`; nonempty `data-turbo-confirm` uses the host confirmation adapter, while empty confirmation is omitted. There is no generated submitter. Only an exact lowercase link `data-turbo-action` is copied: document or destination-Frame `advance` and `replace` enter the shared form history path, exact `restore` fails before request-ID allocation, and a blank, invalid, differently cased, or whitespace-padded value is omitted rather than normalized and may therefore permit destination-Frame inheritance.

Frame destination resolution still starts from the source link: an un-targeted link inside a Frame uses that Frame, its default target may redirect to another active Frame, and explicit named/`_self`/`_parent`/`_top` values use the normal form target rules. The synthetic form itself has no source-Frame origin, so cross-target failures and successes remain owned by the destination. Exact link-owned or inherited destination-Frame `advance`/`replace` actions use the mounted Frame form-history path above and share its restoration scope without a Frame GET; exact `restore` rejects before request-ID allocation. Invalid explicit action metadata is omitted and therefore permits exact destination-Frame inheritance. Scheme, cross-origin, outside-root, excluded-extension, and opted-out links retain the existing browser-equivalent delegation before generated-form interception and ignore the method/Stream metadata. A missing provider or unmounted/historyless action destination rejects only an otherwise eligible generated-form activation. With global generated-form `formMode="off"`, an otherwise eligible same-origin link instead delegates its ordinary URL through the required host `NavigationAdapter.visit(url, "advance")`, ignoring generated form, Frame, action, and confirmation metadata without allocating a request or semantic-link click event; a missing host adapter still fails closed.

Generic cross-document, route-integrated, Frame, nested-container, and history-mutating fragment/anchor behavior; generic host direct-deep-link fragment semantics; concrete host download/non-current browsing-context implementations and device evidence; other non-HTTP scheme policies; Frame preload/previews; browser-hover prefetch; and TTL/request reuse remain unsupported.

## Frame loading and native accessibility

A matching Frame GET response is parsed once and converted into a single-use prepared mutation. Preparation clones the active tree, preserves the exact mounted Frame wrapper, replaces only its candidate children, applies the final canonical response `src`, and rebuilds indexes before any optional final-commit callback can run; a response ID colliding anywhere else in the active document therefore fails before host state can change. The internal mutation primitive can also retarget the logical document URL on the same tree, preserving unrelated node and component identities for the later Frame-history coordinator.

Every ordinary Frame replacement also follows a narrow native counterpart to Turbo 8.0.23's Bardo preservation: current and incoming application-element descendants pair only when each has the same nonblank `id` and raw `data-turbo-permanent` presence. A pair keeps the current node, attributes, complete subtree, state, and disposal registrations, and may move anywhere inside the same incoming Frame; unmarked or one-sided same-ID nodes still structurally remount, with no tag/QName or same-parent requirement. The Frame wrapper and descendants behind protocol wrappers are deliberately not candidates, an outer retained permanent keeps its whole current nested subtree, and a permanent wrapper or ancestor never rejects ordinary replacement. A resulting duplicate active ID rejects the whole prepared mutation because native target lookup remains unique. Parsed response nodes remain a separate immutable tree.

The sole native Frame response morph path is a direct `FrameController.reload()` of a connected, enabled mounted Frame whose current `src` is nonblank and whose current `refresh` is exactly lowercase `morph`. That intent is captured before the request. If the first matching response Frame is reached without an intermediary recurse fetch, the mounted wrapper stays the same object and retains its existing attributes; only the canonical final response URL may update its `src`. Its children undergo one recursively preflighted bounded reconciliation: exact-shape stable-ID ordinary application elements retain identity across ordinary-parent moves within the Frame, incompatible ordinary wrappers may remount around retained stable descendants, and same-parent anonymous exact-QName wrappers use persistent descendant-ID intersection before the state-free ordinal fallback; protocol descendants remount unless a move would reuse an active protocol ID. Stable-ID permanent application descendants use the same opaque rule as Stream morphs: a compatible current node may move anywhere inside the Frame while retaining all current attributes and descendants, a directly omitted permanent remains in place, and incoming-only markers are admitted normally. Mounted and response Frame wrappers, plus every permanent ancestor, remain ineligible. Protocol reparenting, marker-bearing application nodes without stable IDs, incompatible same-ID permanent identities, omitted unmatched permanent-bearing ancestors, duplicate IDs, or out-of-subtree plans reject before any live change, with no structural replacement fallback. Initial/eager/lazy loading, manual `load()`, `setSource()`, same-source `visit()`, link/form/history loads, recurse, Stream Frame replacement targets, and all other document/current-document refresh morph paths remain replacement or unsupported.

During that direct outer reload-morph, an eligible nested Frame remains opaque: its mounted wrapper, attributes, and current children are retained while incoming nested content is ignored. The incoming nested Frame may be absent, or have the same ID with an absent, blank, or URL-equivalent `src`. Eligibility requires the nested Frame's current nonblank `src`, exact lowercase `refresh="morph"`, no permanent ancestor, and the outer Frame as its closest reloadable morph ancestor. After embedded Streams settle and React acknowledges the outer Frame, the controller publishes its render/load lifecycle when configured, then rechecks the exact mounted, connected, enabled child and starts that child controller's own `reload()`. A child can seed its descendant layer only after its own acknowledgement and final recheck. This retains only the bounded nested cascade; it does not promise a browser paint boundary.

An optional shared `FrameLifecycle` exposes a synchronous native `frame-missing` event after an otherwise eligible Expo Turbo XML response and bounded recurse chain fail to produce the expected application Frame. Direct/recurse GETs and Frame-targeted form responses share the same frozen cancellable event. A direct GET or successful form uses its requested destination; a failed cross-target form first transfers from that destination to its exact origin Frame, matching the existing failure-rendering policy. Public detail is limited to that application `frameId`, frozen final primary-response metadata (`redirected`, HTTP `status`, and validated final `url`), and a controlled `visit()` function; the buffered XML body and headers are absent. Stable listener snapshots require an exact synchronous `undefined` return, and failures are redacted to Frame/status metadata. Without prevention, the existing typed `FrameMissingError` remains the default native error surface. Prevention returns the existing explicit `prevented` report and does not pretend content was applied. The package's missing-response path performs no tree mutation before the optional visitor; trusted listeners and that visitor retain their explicit host mutation authority.

`visit({ action })` accepts only `advance` or `replace`, defaults to `advance`, and records the first intent without starting navigation inside the current Frame lease. After dispatch, the loader or form controller rechecks exact request, Frame, history, and proposal ownership; listener-side supersession makes continuation stale and prevents visitor execution. Current prevented work then settles and releases its exact lease before invoking the lifecycle's optional trusted `visitResponse` adapter once with the buffered XML and frozen metadata. This adapter boundary lets a host promote the received response without fetching it again while keeping response bytes out of ordinary event listeners. Calling `visit()` without `preventDefault()` never invokes the adapter; prevention without `visit()` is a handled no-op; a missing or failing requested adapter produces a redacted request error after lease release. Matching Frames, primary `204`, negotiated Turbo Stream responses, wrong MIME, malformed XML, recurse loop/depth failure, absent active Frames, request-lifecycle response prevention, and work already stale, canceled, or superseded before dispatch emit no event. A recurse response that exhausts extraction, including a recurse `204`, emits once for the retained primary response. Browser DOM bubbling, literal `Content missing` markup, raw `Response` exposure, and exact renderer/controller complete timing remain outside this native slice.

An eligible Frame response whose exact sole XML root declares `data-turbo-visit-control="reload"` takes the automatic owning-document response path before matching-Frame extraction, recurse continuation, embedded Stream application, `frame-missing`, or any Frame mutation. Direct and bounded-recurse Frame GETs plus Frame-targeted forms admit nonempty Expo Turbo XML across `2xx`/`4xx`/`5xx`; exact case and value are required, and nested, whitespace-padded, or differently cased values are inert. After rechecking exact Frame, request, form, and history ownership, the controller releases the exact lease and calls the same trusted `visitResponse` adapter with the exact buffered response, `action: "advance"`, and `reason: "visit-control-reload"`. The visitor therefore owns whole-document history, cache, and tree application without a refetch, while the Frame path commits no children, embedded Streams, autofocus, Frame history, or logical document URL. Primary direct responses and handling forms still publish their canonical final Frame `src` on redirect or success; recurse intermediary responses do not retarget the active Frame. Failed cross-target forms transfer application ownership to the exact origin Frame before promotion. An absent or rejected visitor fails closed with a redacted request error, and the promoted result or form terminal report settles only after visitor success; newer ownership cancels stale promotion.

`FrameRequestLoader.load(..., { beforeFrameCommit })` receives one frozen response candidate only after recurse extraction and structural preflight. It runs synchronously under the exact Frame lease and a session-wide commit guard: another document or Frame request cannot claim the session, exact cancellation returns `false`, and any returned value or thenable fails closed with its rejection consumed. Callback failure is redacted and prevents the prepared Frame mutation from beginning; the callback must itself be atomic-on-error and must not mutate the document session or reenter package controllers. Stale, empty, MIME/parse-failed, missing-Frame, and recurse-failed work never enters the callback. The guard clears immediately before the prepared mutation, so synchronous disposal/listener finalization observes committed Frame truth and may start newer work. Lease-backed embedded Stream dispatch then stops undispatched actions with `interrupted: true` after supersession, while exact outer cleanup cannot detach the newer owner. A post-mutation finalization failure raises `FrameCommitError` with frozen committed evidence instead of being mislabeled as an uncommitted request failure. `FrameController` publishes that committed lifecycle as completed while still notifying and rethrowing the finalization error.

The same optional `FrameLifecycle` exposes synchronous native `before-frame-render` after direct/recurse Frame GET or Frame-targeted form XML has matched and structurally preflighted its exact response Frame. Exact Frame ownership blocks package request/controller reentry, and the prepared mutation is revalidated before promoted history or the package mutation can proceed. The frozen logical detail is `{ frameId, newFrame, renderMethod: "replace" | "morph", url, render }`; `newFrame` is the parsed response Frame after embedded Streams are extracted, and `render` is the only mutable slot. Only the bounded direct `FrameController.reload()` path above can report `"morph"`; direct/recurse navigation and Frame-targeted forms remain `"replace"`. Listeners use stable snapshots, return `undefined` synchronously, and may replace `render` with a wrapper that receives only `{ frameId, newFrame, renderDefault }`. The wrapper must call `renderDefault()` exactly once and return `undefined`; that call only admits the package-owned prepared mutation. It cannot pause, cancel, inject an arbitrary native tree, or select a different render mode, and its context does not expose the session. Empty, Stream-only, missing, promoted, malformed, wrong-MIME, canceled, and superseded response paths emit no event. A wrapper failure or no delegation leaves the prepared mutation, embedded Streams, and promoted history unapplied. This is a logical pre-mutation hook, not a browser DOM event or physical paint boundary.

For the bounded direct reload-morph path, `FrameLifecycle` additionally emits notification-only `before-frame-morph` after `before-frame-render` has admitted the default renderer and immediately before child reconciliation. Its frozen `{ currentFrame, frameId, newFrame, url }` detail exposes the exact mounted and parsed response Frame handles without a mutable DOM or session API. A retained nested `refresh="morph"` Frame emits its own event only when the acknowledged parent cascade reaches that Frame's explicit reload. Ordinary connect/load/visit/recurse responses and Frame-targeted forms remain replacements and emit nothing. Listener faults are isolated through the ordinary notification observer surface; prevention and renderer replacement are intentionally unavailable.

`FrameLifecycle` also exposes notification-only native `frame-render` and `frame-load` events. Every controller-owned matching Frame GET prepares a ticket for the exact mounted Frame immediately before mutation, independent of whether `FrameLifecycle` is configured. Once replacement children and embedded Streams settle, only that live Frame at the sealed session revision may acknowledge the ticket and apply Frame autofocus. The controller then publishes terminal GET truth (`busy: false`, `complete: true`) before it dispatches frozen `frame-render` `{ frameId, renderMethod: "replace" | "morph", url }`, rechecks the exact ticket and controller epoch, and dispatches frozen `frame-load` `{ frameId, url }`. A bounded direct outer reload-morph starts its nested cascade only after that acknowledgement and, when configured, those notifications; each candidate rechecks exact mounted/connected/enabled/source/permanent/ancestor state before it reloads. Only the bounded direct reload path can report `"morph"`; applied Frame-target form XML remains `"replace"` and preserves renderer acknowledgement plus the public render/load order without making controller `busy`/`complete` own the form submission. For promoted GETs and successful promoted Frame forms, those notifications precede the synthetic owning-document `before-visit`/`visit`; a committed finalization failure does not repeat them. A `frame-render` listener that reloads, replaces, disposes, or otherwise supersedes the Frame therefore suppresses the stale `frame-load` and cascade. Headless, empty, Stream-only, missing, canceled, superseded, unavailable, and same-ID-replaced Frames emit neither post-render event and do not wait; a failed acknowledgement releases the visual wait without either event. Observer faults are redacted through `FrameLifecycleOptions.onObserverError`; there is no DOM bubbling, browser DOM-attribute, or physical-paint/`nextRepaint()` guarantee.

An optional `FrameHistoryCoordinator` binds the exact session and history ledger plus an optional shared snapshot cache, owning-document visit lifecycle, and navigation adapter. The Frame registry creates an opaque plan before request ownership, optionally captures the outgoing whole document while the mounted Frame still has its prior `src`, and admits only root-visitable explicit or inherited local `advance`/`replace`. Once the exact Frame lease is installed, the requested absolute `src` is published before fetch. A primary redirect publishes its same-origin final URL immediately; a nonredirected successful Expo Turbo XML response does the same before body parsing. That source timing is independent from history commitment, so an empty or failed response may retain the requested/final `src` while leaving Frame children and the logical document URL unchanged.

Turbo 8.0.23 applies Frame action metadata to a lazy source only when the Frame first actually appears. The controller therefore re-reads the exact active Frame, current `src`, and live lowercase `data-turbo-action` at visibility time. Exact `advance`/`replace` prepare the same opaque history plan before `FrameRequestLoader` allocates a request ID or canonicalizes the source, so outgoing snapshots retain the authored prior `src` and the mounted controller shares one restoration scope with later explicit/inherited promotions. Immediate visibility, a synchronous visibility callback, and delayed appearance all return/store the same promoted load. Exact `restore`, missing history coordination, unvisitable sources, or a history/document mismatch enter controller error state once and notify registered error listeners before request ownership; the observer is released and repeated stale visibility callbacks cannot fetch. Blank, invalid, whitespace-padded, and differently cased actions remain ordinary lazy loads. Eager sources, manual `load()`/`reload()`, and source changes after the first load remain history-neutral.

The standalone Expo gallery now supplies a concrete example-owned `VisibilityAdapter` for ordinary Frame layout inside its root and registered nested `ScrollView` regions. Each rendered Frame boundary uses a non-collapsible native anchor and registers a `measureInWindow` callback plus its current clipping-container chain; root and nested scroll views register their window rectangles through the same registry. Visibility is positive intersection with the shared intersection of every cached clipping rectangle, so a Frame that is visible in the root but clipped by an inner scroll region remains hidden. Root/nested layout, scroll, content-size, and window-dimension changes remeasure active geometry. Unknown or unmeasured Frames are hidden, only boolean transitions publish, stale asynchronous Frame/container measurements cannot overwrite newer geometry, and same-ID Frame/container replacement plus final disposal are identity-safe. The gallery also has one horizontal `FlatList` region whose XML declares an ordered explicit Frame-ID row list. Its direct Frame rows require both clipping geometry and the current native `onViewableItemsChanged` full snapshot; the adapter rejects buffered geometry-only rows plus stale key/index, removed, recycled, and replaced rows. Focused fixtures prove the nested and `FlatList` scenarios load exactly once only after admission. Physical-device measurement and arbitrary virtualizer support remain later host/device evidence gates rather than inferred behavior.

One exact mounted Frame controller reuses its restoration identifier; `advance` always pushes, including the same canonical URL, while `replace` preserves the current index. After matching extraction and structural preflight, the guarded commit proposes against the history entry current at response completion. Independent Frames therefore commit in response-completion order instead of invalidating one another. Every admitted whole-document visit or document-targeting form submission advances a session navigation epoch; a promoted Frame captured before that work becomes canceled before it can write history or children, preserving newer document-navigation precedence. If configured, the admission-time outgoing snapshot is stored first; the coordinator then writes history and commits Frame children, final `src`, and the logical document URL as one prepared session mutation. Matching `2xx`/`4xx`/`5xx` XML may promote. `204`, cancellation, MIME/parse/missing-Frame/preflight failure, stale ownership, snapshot failure, and history-host failure change neither history, logical document URL, nor Frame children. Package session and active-tree mutation methods plus destination request-producing controller mutations are blocked during the guarded history write, while exact cancellation is inert. Active protocol nodes, child/attribute collections, and returned Frame/stream-source index arrays are readonly and frozen; package mutation methods update protected backing state instead of exposing writable relationships. Host back/forward remains the existing whole-document snapshot/network restoration path: it uses a retained snapshot when available, otherwise performs one canonical document GET, and never issues a Frame GET. A same-origin primary final URL is retained through this commit, then its late lifecycle is reclassified against the response document's Turbo root: root-visitable URLs emit the synthetic visit, while outside-root or excluded-extension URLs use the coordinator's required host navigation.

A connected Frame exposes its immutable controller snapshot through a host-defined `frameComponent` boundary and `useExpoTurboFrame()`. The hook resolves the nearest connected Frame, so nested Frames receive independent bindings and components outside a Frame receive `undefined`. The default boundary remains a Fragment.

Every ordinary authoritative `DocumentSession` tree generation owns one document-autofocus opportunity, starting at generation zero and advancing on every ordinary `replaceTree()` call, including network visits, document-form responses, refreshes, cached restorations, and an explicit reinstall of the same tree object. A provisional cached-preview generation deliberately clears any pending claim and creates none; its later canonical replacement is a new authoritative generation with its own opportunity. A bounded successful current-document Refresh Stream morph also creates a fresh authoritative generation, but deliberately has no document-autofocus opportunity. Before an autofocus-owning authoritative generation becomes observable, the session records the exact tree, document, ordered stable-ID application-element candidates, and candidate node identities. Candidate traversal follows whole-document order through nested Frames, treats raw `autofocus` presence as opt-in even when authored as `"false"`, excludes protocol wrappers as candidates, and does not enter `template`, Stream, or stream-source subtrees. A parent renderer layout effect claims only the exact rendered generation after descendant native handles mount. React Strict Mode, multiple providers, and provider remounts therefore cannot duplicate the claim; a stale rendered generation cannot consume a newer one; changed candidates or same-ID replacement consume the stale claim without focusing; and unrelated in-generation mutations do not invalidate it.

Document autofocus uses the same optional synchronous `AutofocusAdapter` contract as Frame autofocus: candidates are checked in order, unavailable handles are skipped, and the first available handle is focused. The claim is consumed before adapter lookup, so a missing adapter is a permanent no-op for that generation and late injection cannot steal focus. Throws, invalid return values, Promises, and thenables become redacted `StateError`s through the document render error surface without rolling back the committed logical tree; a host-rendered error boundary may replace its visual subtree. In-place Frame and standalone Stream mutations do not create a new document generation and cannot retrigger this path; the bounded successful current-document Refresh Stream morph creates a generation but intentionally suppresses this path. This contract does not infer browser focusability, cover broader morph autofocus, or claim physical-device behavior.

`ExpoTurboProvider` may also receive an optional synchronous `AutofocusScrollAdapter`. Only after a document, controller-owned Frame, or admitted standalone Stream candidate has successfully focused does the renderer call `canScroll(nodeKey)` and, when that returns `true`, `scrollTo(nodeKey)`. It does not call the scroll adapter for unavailable candidates, preexisting logical Stream focus, missing focus adapters, canceled or stale claims, or an unsuccessful focus attempt. A missing/unavailable scroll adapter consumes no extra work; throws, nonboolean availability, non-`undefined` returns, Promises, and thenables become redacted `StateError`s through the same document or Frame surface without rolling back the committed tree or the completed focus. The package does not discover a scroll container, schedule/retry geometry, or wait for a physical scroll result. A host may keep an accepted request while its own explicit measurements settle. In the document path, existing history and Refresh-scroll effects retain their later ordering; in a Frame path, Frame-boundary autoscroll remains earlier than focus. Nested/virtualized containers, generic anchor mapping, browser behavior, and physical-device parity remain host work.

A separate bounded native-equivalent applies to a completed standalone Stream message: a public direct fragment, a Cable message, or a Turbo Stream MIME form response considers only the first stable authored-ID application candidate carrying raw `autofocus` in Stream/template order from a default exact-target structural `append`, `prepend`, `before`, `after`, `replace`, or `update`; the four insertion actions retain structural semantics with any `method` metadata. Exact-target `replace`/`update method="morph"` are also admitted only when their whole template payload contains no `data-turbo-permanent`. Where the dispatch path has a response-ownership guard, it must still own that response. An earlier candidate retains its claim across later standalone messages before React commits while its exact node identity remains active; removal or identity-changing same-ID replacement discards it, after which a later eligible message may claim its first candidate; an ordinary candidate remains eligible when bounded morph reconciliation retains its exact node identity. The renderer consumes the live identity-bound candidate only after that or a later session revision mounts its descendants. It calls the optional `AutofocusAdapter` only when `getFocusedId()` is present and returns `undefined`; existing logical focus preserves focus, while a missing adapter/snapshot method, unavailable handle, or stale candidate consumes a no-op. Cancellation, supersession, selectors, permanent-bearing or other morphs, custom replacement renderers, no-ops, and document/Frame-embedded Streams create no intent. Native hosts must provide the logical focus snapshot; the package does not synthesize browser IDs, infer DOM focusability, restore existing focus, Turbo Bardo/permanent-node autofocus, select a generic scroll container, wait for browser repaint, or claim physical-device autofocus parity.

After a controller-owned matching Frame GET or a matching Frame-target form XML response commits, the package derives one frozen autofocus candidate list from the exact still-active Frame after its embedded Streams finish. Form response autofocus requires the `FormSubmissionController` to be configured with the exact shared registry that owns the mounted destination Frame. Raw `autofocus` presence is intentional—even `autofocus="false"` opts in—and only ordinary application-element descendants with a stable authored `id` become candidates; the renderer registry and host adapter still decide whether each candidate is renderable and focusable. Traversal follows final document order through nested Frames but excludes every Frame or other protocol wrapper and does not enter `template`, processed Stream, or stream-source subtrees. The internal intent is bound to its originating session, wrapper, ordered candidate keys, and exact candidate node identities and is revalidated when React consumes it. An unrelated Frame may therefore commit without suppressing focus, while headless application, an unmounted Frame, a changed candidate list, or a later same-ID replacement cannot queue stale focus.

`ExpoTurboProvider` accepts an optional synchronous `AutofocusAdapter`. In the parent Frame layout effect, after replacement descendants have registered their native handles, the renderer atomically claims the current internal intent, asks `canFocus(nodeKey)` in order, skips unavailable handles, and calls `focus(nodeKey)` on the first available candidate. The intent is consumed at most once under React Strict Mode. No adapter consumes the intent as a no-op; no focusable candidate or an empty candidate list is also a no-op. Thrown failures, nonboolean `canFocus` results, non-`undefined` `focus` results, Promises, and thenables become redacted `StateError`s through the existing Frame render error surface; the committed Frame remains. A successful focus may then use the shared optional post-focus `AutofocusScrollAdapter` contract above. This Frame-specific contract does not map XML `autofocus` to React Native `autoFocus`, infer arbitrary platform focusability or scroll containers, or cover direct headless Frame application, standalone Stream/morph mutations, or physical-device behavior.

Frame autoscroll is a separate one-shot render effect. A controller-owned matching Frame GET or matching Frame-target form XML response requests it when either the mounted Frame or matching response Frame has raw `autoscroll` presence, including `autoscroll="false"`; its frozen request always reads only the mounted wrapper's exact `data-autoscroll-block` (`start`, `center`, `end`, or `nearest`, default `end`) and `data-autoscroll-behavior` (`smooth`, otherwise `auto`). At the same parent Frame layout boundary, after replacement children and embedded Streams mount and before autofocus, the renderer calls the optional synchronous `FrameAutoscrollAdapter` only for the exact registered mounted `frameId` and only when `canScroll(frameId)` is `true`. No adapter or unavailable boundary permanently consumes that render's intent as a no-op, so a later adapter cannot scroll stale content. Smooth is a start request, not a physical-completion promise. Thrown failures, nonboolean availability, non-`undefined` scroll results, Promises, and thenables are redacted through the existing Frame render error surface without rolling back the committed Frame. This is a Frame-boundary native equivalent; it does not infer arbitrary scroll containers/nodes or cover headless application, standalone Stream/morph changes, browser first-child semantics, or physical-device evidence.

`busy` becomes true immediately while a Frame GET owns the controller and false on every terminal path. For a mounted matching GET replacement, the terminal publish waits for the renderer acknowledgement and autofocus attempt; unmounted/headless and nonreplacement terminal paths settle immediately. `complete` means only that the Frame is not currently loading; it is true after success, an empty `204`, failure, or cancellation. Hosts must inspect `status` and `hasBeenLoaded` when the terminal result matters. This slice covers Frame GET activity only. Document visits, targeted form submissions, delayed visual progress, submitter state, and Frame-specific live announcements remain separate contracts.

The binding includes a frozen `{ busy }` accessibility state for a host to pass to an actual native boundary. The package does not import React Native, infer that a disabled Frame makes all descendants inaccessible, announce server-authored status, or mutate `busy`, `complete`, or `aria-busy` into the logical XML tree. Controller-state changes preserve the boundary and current children; only boundary or nearest-Frame consumers rerender until a protocol mutation actually replaces children. Host boundary failures use the same registered Frame error surface as other protocol nodes.

## Programmatic Frame visits

`FrameControllerRegistry.visit(url, { frame, ... })` resolves through the shared interaction-style Frame target rules. A credential-free, fragment-free same-origin HTTP(S) URL targeting a Frame reuses that Frame's controller and loader; visiting its current `src` without an action reloads it without applying the document root policy. An omitted action inherits only an exact lowercase `advance`, `replace`, or `restore` from the resolved destination Frame; invalid values are ignored, and `action: null` explicitly masks inheritance for a present Turbo non-action. With a configured `FrameHistoryCoordinator`, local `advance`/`replace` require root visitability and use the destination Frame controller's stable history scope. Local `restore` rejects because restoration is whole-document-only. Promoted, `_top`, and external visits do not inherit. When the registry receives the shared document controller, a same-origin top/promoted target enters its root-aware path for `advance` or history-backed `replace`/`restore`; a historyless `replace`/`restore` rejects there. The compatibility fallback directly awaits `NavigationAdapter.visit` and does not claim root-aware document ownership. A credential-free cross-origin HTTP(S) URL awaits `openExternal`. Adapter failures reject the visit without changing Frame request or source ownership.

URL admission happens before target dispatch, any host-adapter call, or any Frame request/source ownership change. Invalid, non-HTTP(S), credential-bearing, fragment-bearing, or invalid-document URLs fail as redacted `TargetError`s and invoke neither navigation nor fetch, so a rejected visit cannot cancel or replace an already-loading Frame source.

Root-visitable registered XML links without generated-form metadata call this exact API for nearest-Frame and active named-Frame capture. Generated-form links use the controller above and share form-submission rather than Frame GET ownership. Disabled presence ignores before either path; non-current browser targets and downloads bypass both paths through the explicit host link adapter or fail closed when it is absent. Present non-action values mask destination-Frame inheritance only on the ordinary link path, while browser-bypassed action and generated-form metadata retain their delegated path. With no link-owned action, the resolved destination Frame's exact action is admitted through the ordinary Frame registry. Exact local `advance`/`replace` uses Frame history when configured, and generated-form `advance`/`replace` reuses the same mounted controller scope through form history; `_top` promotion remains document-owned and Frame `restore` remains unsupported. `data-turbo-confirm` alone is inert during plain activation; on a generated-form link a nonempty value uses the injected confirmation adapter. Authority-form `mailto:`/`tel:` URLs, empty `tel:` subscribers, fragments, raw C0/DEL controls, credentials, `javascript:`, `data:`, `file:`, `blob:`, `sms:`, and arbitrary external schemes remain rejected by fetch paths. XML form capture, fragment/anchor behavior, and generic host router mapping remain separate contracts.
